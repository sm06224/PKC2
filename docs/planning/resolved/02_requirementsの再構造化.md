# 02. requirements の再構造化

requirements の内容を「要求」「設計方針」「実装方式案」に分離し、機能カテゴリ別に再構造化する。

---

## 2.1 プロダクト目的

> **PKC Gen2 は、成果物・知識・時間を一体として扱う「時間認識型知識コンテナ」である。**

- 単なる記録の集積や知識の整理にとどまらず、時間と成果を内包した知識・活動管理を行う
- 保存する箱から、時間とともに進化する知識・成果の活動体へ
- 管理するためのツールではなく、成果物と知識を迎えに行くための器

**これは要求**: PKC2のアイデンティティを定義する最上位の要求。

## 2.3 コア機能要件

### 2.3.1 統一レコード構造【要求】
- すべての情報はRecord（LID/title/body/created_at/updated_at）���して保持する
- データ構造はUIや用途に依存しない
- 根拠: Ch.4, Ch.5

### 2.3.2 Archetype（類型）システム【要求】
- Recordのbodyをどう解釈するかを外部から定義するしくみ
- データ構造を変更せず、解釈・表示・操作ルールのみを規定
- 初期類型: text, textlog, todo, form, attachment, generic, opaque
- 根拠: Ch.5, Ch.6

### 2.3.3 Relation（リレーション）モデル【要求】
- Record間の関係をfrom/to/kind/timestampで表現
- 種別: structural（フォ���ダ等）, categorical（タグ等）, semantic（参照等）, temporal（時間順序）
- フォルダ・タグ・グラフを単一モデルで統合
- 根拠: Ch.7

### 2.3.4 Revision（リビジョン）管理【要求】
- 非破壊改訂: 既存データを上書きせず、変更はRevisionとして追加
- 時系列で連結し、任意時点の状態再現を可能とする
- 根拠: Ch.4.4, Ch.9.5

### 2.3.5 時間モデル【要求】
- 全オブジェクト(Record/Revision/Relation/Asset)に時間情報を付与
- 単一時間軸で管理し、時系列処理を構造的に可能とする
- 根拠: Ch.4.7, Ch.9

### 2.3.6 Asset管理【要求】
- バイナリデータはRecordから分離して管理
- Recordから参照される形で利用
- 根拠: Ch.4.6

## 2.4 補助機能要件

### 2.4.1 ヒストリカルビュー【要求（段階的）】
- グローバルタイムライン
- レコード中心ヒストリ
- 構造ヒストリ
- 成果物到達ヒストリ
- 根拠: Ch.10

### 2.4.2 逆算型進捗管理【設計思想 + 要求】
- 成果物予約型TODOを起点とした構造化
- 進捗 = 未来の成果物と現在の差分
- 根拠: Ch.11

### 2.4.3 活動量ダッシュボード【将来要件】
- 既存データの集計結果として導出
- 自己把握・振り返り目的
- 根拠: Ch.12

### 2.4.4 テンプレートとクローン【将来要件】
- 3種クローン（完全/構造/成果物予約）
- テンプレート適用は一方向（同期なし）
- 根拠: Ch.14
