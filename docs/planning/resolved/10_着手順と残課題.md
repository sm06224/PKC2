# 10. 最初の実装着手順・残課題・判断保留事項

## 10.1 最初の実装着手順

Phase 0 の最小タスクとして、以下の順で着手する。

### Step 1: プロジェクト基盤（目安: 1セッション）
1. `package.json` 作成（TypeScript, Vite, Vitest）
2. `tsconfig.json` 作成
3. `build/vite.config.ts` 作成（単一HTMLビルド設定）
4. `src/index.html` + `src/main.ts` 作成（空のエントリポイント）
5. `npm run dev` / `npm run build` / `npm test` が動作することを確認

### Step 2: コアデータモデル型定義（目安: 1セッション）
1. `src/core/model/record.ts` — Record型、LID生成、ファクトリ
2. `src/core/model/relation.ts` — Relation型、RelationKind
3. `src/core/model/revision.ts` — Revision型
4. `src/core/model/asset.ts` — Asset型
5. `src/core/model/container.ts` — Container型
6. `tests/core/record.test.ts` — 基本テスト

### Step 3: Archetypeインターフェース（目安: 1セッション）
1. `src/core/archetype/archetype.ts` — Archetypeインターフェース
2. `src/core/archetype/text.ts` — text Archetype最小実装
3. `tests/core/archetype.test.ts` — テスト

### Step 4: 基礎ドキュメント（Step 1-3と並行可能）
1. `docs/design/vision.md`
2. `docs/design/domain-model.md`
3. `docs/design/glossary.md`

## 10.2 次の一手として実装着手すべき最小タスク

> **Step 1: プロジェクト基盤セットアップ**

具体的には:
```bash
npm init -y
npm install -D typescript vite vitest
```
に続いて、`tsconfig.json`、`vite.config.ts`、`src/main.ts`、`src/index.html` を作成し、
`npm run build` で `dist/pkc2.html`（空だが動く単一HTML）が生成される状態にする。

これが「PKC2のコードが1行でも書ける状態」の最小条件である。

## 10.3 残課題・判断保留事項

### 判断が必要な事項

| # | 事項 | 選択肢 | 推奨 | 理由 |
|---|------|--------|------|------|
| J-1 | UIフレームワーク | (a) Vanilla TS (b) Preact (c) Lit (d) Solid | **(c) Lit** または **(a) Vanilla TS** | 単一HTMLサイズ制約。Preact/Solidでも可だがバンドルサイズ注意 |
| J-2 | 状態管理 | (a) 自前Store (b) Zustand (c) 信号ベース | **(a) 自前Store** | 外部依存最小化。PKC2のコアが小さいうちは自前で十分 |
| J-3 | CSS戦略 | (a) グローバルCSS (b) CSS Modules (c) CSS-in-JS | **(a) グローバルCSS** | 単一HTML制約。PKC1と同じアプローチが最もシンプル |
| J-4 | 時間表現 | (a) unix ms (b) ISO 8601 (c) 両方保持 | **(b) ISO 8601** | requirementsはunix時刻と言っているが、ISO 8601の方が人間可読。内部はDate.now()ミリ秒、表示・保存はISO |
| J-5 | LID生成方式 | (a) UUID v4 (b) ULID (c) nanoid | **(a) UUID v4** | PKC1互換。crypto.randomUUID()で生成 |
| J-6 | Revision保存方式 | (a) 全状態スナップショット (b) 差分 | **(a) 全状態** | 初期はシンプルに全状態保存。差分最適化は後 |
| J-7 | Markdownパーサー | (a) marked (b) markdown-it (c) 自前最小 | **(c) 自前最小** → Phase 2で(a)へ | 初期はcode block + heading + bold/italic + list のみ |

### 未解決の技術課題

| # | 課題 | 影響 | 対応方針 |
|---|------|------|---------|
| T-1 | 単一HTMLのサイズ上限 | PKC1は圧縮前1.1MB。データ含めるとさらに増大 | ビルド最適化。CDN progressive enhancement |
| T-2 | IDBのストレージ上限 | ブラウザにより異なる（通常50MB〜） | 警告表示。OPFS検討（Phase 2） |
| T-3 | File System Access APIの対応状況 | Chrome系のみ | フォールバック: ダウンロードベース |
| T-4 | Asset base64のサイズ効率 | base64は33%増 | 圧縮（gzip/brotli）。OPFS検討 |
| T-5 | Rehydrateの信頼性 | HTML解析で復元するため壊れやすい | 構造化データ領域を明確に区切る設計 |

### 設計上の未決事項

| # | 事項 | 方向性 |
|---|------|--------|
| D-1 | Relationの削除は論理削除か物理削除か | 非破壊原則からは論理削除が望ましいが、初期は物理削除でもよい |
| D-2 | Recordの「削除」はどう表現するか | 削除フラグ(soft delete) vs Revision with empty body vs 実削除 |
| D-3 | Container.metaに何を持つか | PKC1はgroups/manual_order/quick_access等を持っていた。PKC2ではRelation正規化により大幅に減るはず |
| D-4 | Archetype登録はビルド時固定かランタイム動的か | 初期はビルド時固定。将来的にはプラグイン化を検討 |
| D-5 | 複数コンテナ間の関係をどう扱うか | requirementsスコープ外だが、将来拡張として設計に余地を残す |
