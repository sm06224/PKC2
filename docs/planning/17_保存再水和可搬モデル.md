# 17. 保存・再水和・可搬モデル

本文書は PKC2 のデータが「どこに・どう存在するか」を定義する上位設計文書である。
attachment 単独の問題ではなく、Workspace / Portable HTML / ZIP / Template の
4系統を統一的に整理し、保存・再水和・可搬の設計基盤を固定する。

**ステータス**: 設計固定（実装は後続 Issue で段階的に進める）

---

## 17.1 目的と背景

### 目的

PKC2 の存在形態を4系統に分類し、各系統間の変換契約を定義する。
これにより以下を達成する:

1. 日常作業（Workspace）と可搬成果物（HTML/ZIP/Template）の責務境界を明確にする
2. 軽量配布と完全アーカイブを両立する export モデルを確立する
3. attachment の body 肥大化問題を body/assets 分離で解決する設計根拠を与える
4. 将来の ZIP・Template 拡張に対して、今から設計の余地を確保する

### 背景

Issue #33 で attachment archetype の最小検証が完了した時点で、以下の課題が顕在化した:

1. **body 肥大化**: `entry.body` に base64 実データを直格納しており、sidebar/filter/sort の軽量処理に不要な重いデータが混入する
2. **export 単一モード**: HTML export が実質「全部入り」前提で、軽量配布と完全アーカイブを選択できない
3. **assets 未活用**: `Container.assets` が型定義済みなのに未使用で、データ分離の設計余地が放置されている
4. **可搬形式の未整理**: ZIP やテンプレートを後付けで追加すると Portable HTML と契約が競合しやすい

本文書はこれらの課題に対し、実装に先立って上位モデルを固定する。

## 17.2 4系統モデル

PKC2 のデータは「どこに・どう存在するか」で4系統に分かれる。

### 概念マップ

```
┌─────────────────────────────────────────────────────────┐
│                    PKC2 Data Universe                    │
├──────────────┬───────────────┬─────────────┬────────────┤
│  Workspace   │ Portable HTML │ Portable    │ Template   │
│  (Browser)   │ (Single .html)│ Package     │ Package    │
│              │               │ (.pkc2.zip) │ (.pkc2t)   │
├──────────────┼───────────────┼─────────────┼────────────┤
│ IDB 分割保存  │ 自己完結      │ 完全データ   │ 構造のみ   │
│ リアルタイム  │ 配布可能      │ 高圧縮      │ データなし  │
│ 大容量対応   │ 軽量/完全選択 │ 機械処理可能 │ 再利用可   │
└──────────────┴───────────────┴─────────────┴────────────┘
```

### (A) Workspace — ブラウザ内作業環境

| 項目 | 内容 |
|------|------|
| **目的** | ユーザーの日常編集環境。データの一次保存先 |
| **保存形態** | IndexedDB（将来的に cid + lid 分割） |
| **制約** | ブラウザ依存。ストレージ quota。同一 origin 内 |
| **ユースケース** | 開く → 編集 → 自動保存 → 再アクセス |
| **ライフサイクル** | HTML を開く → 再水和（Rehydrate） → IDB に接続 → 以降 IDB が source of truth |

### (B) Portable HTML — 配布可能な単一 HTML

| 項目 | 内容 |
|------|------|
| **目的** | 可搬性のある自己完結ドキュメント。開けばそのまま動く |
| **保存形態** | 単一 `.html` ファイル（`#pkc-data` にデータ埋め込み） |
| **制約** | ファイルサイズ（ブラウザメモリ）、base64 膨張、推奨 10 MB 以下 |
| **ユースケース** | メール添付、USB 配布、GitHub Pages、オフライン閲覧 |
| **モード** | editable / readonly-rehydratable × light / full（17.3 参照） |

### (C) Portable Package — ZIP 可搬パッケージ

| 項目 | 内容 |
|------|------|
| **目的** | 大容量データの効率的なバックアップ・移行 |
| **保存形態** | `.pkc2.zip`（`container.json` + `assets/` ディレクトリ） |
| **制約** | PKC2 アプリで import が必要。単独では開けない |
| **ユースケース** | フルバックアップ、マシン間移行、バージョン管理（Git 等）、CI 連携 |
| **サブタイプ** | PKC2 Package（完全再現）/ Document Set（best-effort 再構成） |

### (D) Template Package — 構造テンプレート（将来契約）

| 項目 | 内容 |
|------|------|
| **目的** | Container 構造・レイアウト・テーマの再利用。データは含まない |
| **保存形態** | `.pkc2-template.zip`（schema + archetype 定義 + スタイル + サンプル構造） |
| **制約** | ユーザーデータを含んではならない |
| **ユースケース** | プロジェクトテンプレート配布、AI 生成テンプレート、組織標準テンプレート |

> **注意**: Template Package は概念定義に留める。近い Issue で実装する前提にしない。
> 面白いが別系統の大きなテーマであり、attachment/export 戦略と混同しやすいため、
> 設計の整理としてここに位置づけるが、実装は必要性が確認されてから行う。

### 系統間の関係

```
Workspace (IDB)          ← 一次保存先、常時利用
  ↕ rehydrate / export
Portable HTML (.html)    ← 配布可能な自己完結成果物
  ↕ import / export
Portable Package (.zip)  ← 大容量・高効率の可搬形式
  ↕ apply
Template Package (.zip)  ← 構造のみの再利用テンプレート（将来）
```

## 17.3 Portable HTML の mode

### 2軸 × 2軸 = 4 モード

|  | **Light** | **Full** |
|---|---|---|
| **Editable** | テキスト系のみ。attachment はメタ参照。即編集可 | 全データ圧縮埋め込み。完全自己完結。重い |
| **Readonly-Rehydratable** | 閲覧のみ。再水和で Workspace に昇格可 | 閲覧のみ。再水和で完全復元 |

### 各モードの詳細

#### Editable + Light（デフォルト export）
- **用途**: 日常的な共有・配布
- **`#pkc-data` 内容**: entries（テキスト系 body は完全含む）、relations、revisions。**assets は空**
- **attachment entries**: body にメタデータ（name, mime, asset_key）のみ。**data は含まない**
- **サイズ**: テキスト中心なら数十 KB
- **import 時**: attachment data が欠落していることを検知し、UI で「同梱対象外」として通知

> **重要: Light は「欠落」ではなく「同梱対象外」**
> attachment entry 自体とメタデータは残る。data 実体は最初から同梱しない設計である。
> import/rehydrate しても自動復元されない。UI 上は「同梱されていない asset」として扱う。

#### Editable + Full
- **用途**: 完全アーカイブ、オフライン作業
- **`#pkc-data` 内容**: entries、relations、revisions、**assets 含む（圧縮済み）**
- **サイズ**: attachment 量に比例（数 MB 〜数十 MB）
- **圧縮**: asset 単位で gzip + base64（17.7 参照）

#### Readonly-Rehydratable + Light
- **用途**: プレゼンテーション、レポート配布
- **`#pkc-data` 内容**: Editable + Light と同じ
- **`pkc-meta` に `readonly: true` フラグ**
- **UI**: 編集ボタン非表示。「再水和」ボタンで Workspace に昇格
- **再水和**: IDB にデータを書き込み、editable モードで再レンダリング

#### Readonly-Rehydratable + Full
- **用途**: 完全アーカイブの配布（編集は意図しないが復元可能）
- **`#pkc-data` 内容**: Editable + Full と同じ
- **再水和**: IDB に全データ（assets 含む）を書き込んで Workspace 化

> **重要: readonly-rehydratable は権限制御ではない**
> - セキュリティ境界ではない。HTML を書き換えれば bypass 可能
> - artifact 上の UI ポリシー / 起動ポリシーである
> - rehydrate で editable workspace に昇格できる
> - 「本当の閲覧専用保護」ではないことを設計者・利用者の双方が理解する前提

### Subset Export（将来）

| 対象 | 方法 |
|------|------|
| **全体** | Container 全体を export |
| **entry 単体** | 指定 entry + 関連 relations + 参照先 tag |
| **folder（structural）** | structural relation で子要素を再帰収集 |
| **selection** | UI で選択した複数エントリ |
| **archetype filter** | 特定 archetype のみ（例: todo のみ export） |

**Relation の扱い（段階導入）**:
- **Phase 1**: export 対象に含まれる entry 間の relation のみ含める。片方が対象外の relation は除外
- **Phase 2（将来）**: 外部参照を `external_refs` として記録し、import 時に情報保持

> **注意**: subset export の relation 扱いは段階導入にする。
> 初手から `external_refs` まで固めると scope が大きくなりすぎるため、
> まずは内部完結 relation のみで subset export を早く成立させる。

## 17.4 Attachment の body/assets 分離

### 現行と提案の比較

```
【現行（Issue #33 時点、暫定）】
entry.body = '{"name":"doc.pdf","mime":"application/pdf","data":"<巨大base64>"}'

【提案】
entry.body = '{"name":"doc.pdf","mime":"application/pdf","size":102400,"asset_key":"ast-xxxx"}'
container.assets["ast-xxxx"] = "<base64 data>"
```

### body と assets の責務分離

| 項目 | body（メタデータ） | assets（実データ） |
|------|-------------------|-------------------|
| **責務** | ファイル名・MIME・サイズ等の表示情報 | 実際のバイナリデータ（base64） |
| **ロード** | 常時（sidebar, filter, sort に必要） | 遅延（detail view 表示時のみ） |
| **サイズ** | 数百 bytes | KB〜MB |
| **export light** | 含む | 含まない |
| **export full** | 含む | 圧縮して含む |

### asset_key 参照モデル

```typescript
// entry.body (attachment archetype) — 提案形式
interface AttachmentBodyV2 {
  name: string;
  mime: string;
  size: number;           // decoded byte size（表示用）
  asset_key: string;      // container.assets のキー
  encoding?: 'base64' | 'gzip+base64';  // asset の符号化方式
}

// container.assets
{
  "ast-abc123": "<base64 data>",
  "ast-def456": "<gzip+base64 data>"
}
```

- `asset_key` は `"ast-" + generateLid()` で生成
- entry と asset は 1:1 対応（将来的に `asset_keys: string[]` で複数化可能）
- `asset_key` が null/undefined の場合は「ファイル未添付」

### Backward Compatibility

```
parse 時の優先順位:
1. body に asset_key があれば → assets[key] からデータ取得
2. body に data があれば → 旧形式として直接利用（fallback）
3. どちらもなければ → 空 attachment

migration 方針:
- 明示的な一括 migration action は不要
- 旧形式 body を読んだ場合、次回 save 時に新形式で保存（lazy migration）
- import 時に旧形式を検出 → assets に移行 → body を参照形式に変換
```

### パフォーマンス改善効果

| 操作 | 現行（暫定） | 改善後 |
|------|-------------|--------|
| sidebar render（100 entries, 10 attachments × 500KB） | ~65 MB の JSON parse | ~100 KB の JSON parse |
| archetype filter | 全 body を走査 | 軽量 body のみ走査 |
| IDB save（debounce 300ms） | Container 全体書き込み | entries のみ更新。assets は変更時のみ |
| HTML Light export | 不可能（全埋め込み） | 可能（assets 除外） |

> **注意**: `Container` 型自体は変更不要。`assets: Record<string, string>` は既に定義済み。
> 未活用だったフィールドを設計通りに使い始めるだけである。

## 17.5 IDB 保存モデルの段階進化

### 現行モデル

```
DB: pkc2
  └── Object Store: containers
        ├── key: "<container_id>" → value: Container (丸ごと)
        └── key: "__default__"    → value: "<container_id>"
```

### Phase 1: Assets 分離（最小変更）

```
DB: pkc2 (version 2)
  ├── Object Store: containers
  │     ├── key: "<cid>" → value: { meta, entries, relations, revisions }  ← assets 除外
  │     └── key: "__default__" → value: cid
  └── Object Store: assets (new)
        └── key: "<cid>:<asset_key>" → value: base64 string
```

| 項目 | 内容 |
|------|------|
| **変更量** | 小。ContainerStore interface に asset 操作を追加 |
| **効果** | save debounce で assets が毎回書き込まれない |
| **互換** | 旧 DB (version 1) → onupgradeneeded で migration |
| **保存** | container.assets を空にして containers store に保存。assets は別途 |
| **読込** | container load 後に必要な assets を lazy load |

**Phase 1 で追加する ContainerStore interface:**

```typescript
interface ContainerStore {
  // 既存
  save(container: Container): Promise<void>;
  load(id: string): Promise<Container | null>;
  loadDefault(): Promise<Container | null>;
  delete(id: string): Promise<void>;

  // Phase 1 追加
  saveAsset(cid: string, key: string, data: string): Promise<void>;
  loadAsset(cid: string, key: string): Promise<string | null>;
  deleteAsset(cid: string, key: string): Promise<void>;
  listAssetKeys(cid: string): Promise<string[]>;
}
```

### Phase 2: 完全分割（将来、実測後に判断）

```
DB: pkc2 (version 3)
  ├── meta       └── <cid> → ContainerMeta
  ├── entries    └── <cid>:<lid> → Entry
  ├── relations  └── <cid>:<rid> → Relation
  ├── revisions  └── <cid>:<rev_id> → Revision
  ├── assets     └── <cid>:<asset_key> → string | ArrayBuffer
  └── workspace  └── __default__ → cid
```

| 項目 | 内容 |
|------|------|
| **変更量** | 大。ContainerStore interface 全面改修、persistence listener 粒度変更 |
| **効果** | entry 単位の読み書き。大量エントリでもスケール |
| **注意** | Container の immutable 操作モデルとの整合が課題 |

### Phase 判断基準

| 基準 | Phase 1 で十分 | Phase 2 が必要 |
|------|---------------|---------------|
| エントリ数 | ~500 以下 | 1000+ |
| attachment 容量 | ~50 MB 以下 | 100 MB+ |
| save 頻度 | debounce 300ms で問題なし | entry 単位の差分保存が必要 |

**推奨: Phase 1 を先行実装。Phase 2 は実測データに基づいて判断。Phase 2 を「確定」とはしない。**

## 17.6 Export 形式

### HTML Export 契約

| モード | pkc-data 内容 | assets | サイズ目安（100 entries, 10 files × 500KB） |
|--------|-------------|--------|---------------------------------------------|
| **HTML Light Editable** | entries(body軽量) + relations + revisions, `assets: {}` | なし | ~200 KB |
| **HTML Full Editable** | entries + relations + revisions + assets（圧縮base64） | 全埋め込み | ~4 MB |
| **HTML Light Readonly** | 同 Light + `data-pkc-mode="readonly"` | なし | ~200 KB |
| **HTML Full Readonly** | 同 Full + `data-pkc-mode="readonly"` | 全埋め込み | ~4 MB |

**pkc-data 構造の拡張:**

```json
{
  "container": { "meta": {}, "entries": [], "relations": [], "revisions": [], "assets": {} },
  "export_meta": {
    "mode": "editable | readonly",
    "scope": "full | light",
    "asset_encoding": "base64 | gzip+base64",
    "exported_at": "2026-04-07T...",
    "source_cid": "xxx"
  }
}
```

- `export_meta` は pkc-data のトップレベルに追加。import 時にどのモードで export されたかを判別
- `mode` と `scope` の組み合わせで4モードを表現

### ZIP PKC2 Package（完全再現）

```
my-container.pkc2.zip
├── container.json          # ContainerMeta + entries + relations + revisions (assets: {})
├── assets/
│   ├── ast-xxxx.bin        # 生バイナリ（base64 decode 済み）
│   ├── ast-yyyy.bin
│   └── ...
└── manifest.json
    {
      "format": "pkc2-package",
      "version": 1,
      "created_at": "...",
      "entry_count": 100,
      "asset_count": 10,
      "total_size": 5242880
    }
```

- assets は **生バイナリ** で保存（base64 膨張なし + zip deflate で高圧縮）
- import 時に base64 encode して IDB に格納

### ZIP Document Set（best-effort 再構成）

```
my-documents.pkc2-docs.zip
├── entries/
│   ├── entry-title-1.md     # text → markdown
│   ├── entry-title-2.json   # todo/form → JSON
│   └── ...
├── attachments/
│   ├── doc.pdf              # 元のファイル名で保存
│   └── ...
├── relations.json
└── manifest.json
```

- 汎用ファイルシステムとして読める形式
- PKC2 への re-import は best-effort（完全再現は保証しない）
- 外部ツールとの連携用

### Template Package（将来契約）

- 構造定義のみ。ユーザーデータを含まない
- archetype 設定、relation schema、CSS theme、空 entries のスケルトン
- 今回は概念定義のみ。実装は将来の別 Issue で行う

### Import の拡張

| 入力 | 処理 |
|------|------|
| `.html`（現行） | DOMParser → pkc-data 抽出 → Container |
| `.html`（新: export_meta 付き） | 同上 + asset_encoding に応じた decode |
| `.pkc2.zip` | ZIP 解凍 → container.json + assets/ → Container |
| `.pkc2-docs.zip` | ZIP 解凍 → best-effort Entry 生成 |

## 17.7 圧縮方針

### 方式比較

| 方式 | テキスト系 圧縮率 | バイナリ系 | 外部依存 | ブラウザ API |
|------|-----------------|-----------|---------|-------------|
| **base64**（現行） | なし（+33%膨張） | なし（+33%膨張） | なし | `btoa/atob` |
| **gzip + base64** | 30-55% 削減 | ~3% 削減 | なし | `CompressionStream` |
| **gzip + base32768** | 40-60% 削減 | ~15% 削減 | **あり**（npm） | `CompressionStream` + lib |

### 評価

| 評価軸 | base64 | gzip+base64 | gzip+base32768 |
|--------|--------|-------------|----------------|
| サイズ効率（テキスト系） | × | ○ | ◎ |
| サイズ効率（バイナリ系） | × | △ | △ |
| 実装コスト | ◎ | ○ | △ |
| PKC2 思想（外部依存最小） | ◎ | ◎ | **×** |
| fallback 設計 | — | base64 に fallback | 複雑 |
| デバッグ容易性 | ◎ | ○ | × |

### 採用方針

**gzip + base64 を採用。base32768 は不採用。**

理由:
1. `CompressionStream` は外部依存なしで利用可能（Chrome 80+, Safari 16.4+）
2. テキスト系データで 30-55% の削減は十分な効果
3. 画像等の既圧縮データでは効果が薄いが、膨張もしない
4. fallback が単純（`CompressionStream` 未対応 → 非圧縮 base64）
5. base32768 は追加 10-15% の改善のために npm 依存を増やす — PKC2 の「外部依存最小」思想に反する

### 格納先別の圧縮方式

```
IDB:          base64（非圧縮。CPU コスト回避、ストレージに余裕あり）
HTML Full:    gzip+base64（サイズ削減が直接メリット）
ZIP Package:  生バイナリ + zip deflate（最高効率）
```

- asset 単位で圧縮適用
- `asset_encoding` フィールドで asset ごとに方式を記録

## 17.8 Backward Compatibility 方針

### attachment body 形式の互換

| 形式 | 判定方法 | 動作 |
|------|---------|------|
| **新形式**（asset_key 参照） | `body.asset_key` が存在 | `assets[key]` からデータ取得 |
| **旧形式**（data 直格納） | `body.data` が存在 | そのまま利用。次回 save 時に新形式へ lazy migration |
| **空** | どちらもなし | 空 attachment として表示 |

- 旧形式の PKC2 HTML を import しても即座に動作する
- migration は save 時に暗黙的に行われる（明示的な一括 migration は不要）
- 旧形式 → 新形式への変換は不可逆だが、データ損失はない

### export_meta の互換

- `export_meta` が存在しない HTML は「旧形式 editable-full」として扱う
- 新しい export_meta を持つ HTML を旧バージョンの PKC2 で開いた場合:
  - `export_meta` は無視される（pkc-data.container だけ読む既存ロジック）
  - 大きな非互換は発生しないが、asset_key 参照の attachment は表示できない

### IDB version の互換

- version 1 → 2: `onupgradeneeded` で `assets` object store を作成
- 既存 container 内の `assets` は次回 load 時に assets store に移行
- downgrade は非対応（version 2 の DB を version 1 で開くことはない想定）

### Phase 分割

```
Phase α（設計固定）:     A                          ← 本文書で完了
Phase β（基盤改修）:     B → C → D                  ← attachment 問題の解決
Phase γ（export 拡張）:  E → F → G                  ← 多層 HTML の完成
Phase δ（可搬形式）:     H → I → J                  ← ZIP / Template（将来）
```

- **Phase α + β で現行の attachment body 肥大化問題はすべて解決される**
- Phase γ で「多層可搬知識コンテナ」の HTML 側が完成する
- Phase δ は必要性が確認されてから着手する

### 制約の遵守確認

| 制約 | 本設計での遵守 |
|------|-------------|
| 5層構造を壊さない | Container 型は core 不変。assets 操作は core/operations に追加可。IDB/export は adapter/platform |
| core に browser API を入れない | asset_key 生成は既存 `generateLid()` を使用。CompressionStream は adapter 層のみ |
| renderer/binder/presenter 境界 | presenter は assets lookup を adapter 経由で行う。renderer/binder は変更なし |
| 単一 HTML 契約 | builder が生成する shell.html 構造は維持。pkc-data の中身が拡張されるだけ |
| dist は正本扱いしない | dist は builder の出力。設計は docs/planning、実装は src/ が正本 |
