# 06. 初期開発スコープ

requirements をそのまま全部実装対象にせず、3段階に分解する。

---

## Phase 0: Project Bootstrap

### 目的
PKC2の開発基盤を整備し、最初のコードが書ける状態にする。

### 含めるもの
- TypeScript + ビルド環境（Vite）のセットアップ
- ディレクトリ構成の確定
- コアデータモデルの型定義（Record, Relation, Revision, Asset, Container）
- Archetype インターフェースの型定義
- 最小テスト環境（Vitest）
- ドキュメント構成の確定
- 用語集（glossary）初版
- ドメインモデル文書初版

### 含めないもの
- UI実装
- 永続化
- Export/Import
- 具体的なArchetype実装

### requirements対応
- Ch.4（データモデル）: 型定義として反映
- Ch.5（Archetype概念）: インターフェース定義として反映
- Ch.7（Relation）: 型定義として反映

### 完了条件
- `npm run build` でエラーなく単一HTMLが生成される（中身は空でよい）
- `npm test` でコアモデルの型テストが通る
- ドメインモデル文書が docs/ に存在する

### 想定リスク
- ビルド設定の試行錯誤で時間を消費する
- データモデルの型定義が早期に確定しきれない可能性

---

## Phase 1: Core Data + Minimal UI

### 目的
**requirements を満たす最小の成立核を実装する。**

「Recordを作成し、Relationで関連付け、Revisionで履歴を保持し、
単一HTMLとして保存・再読み込みできる」状態。

### 含めるもの（厳選）

| # | 機能 | 詳細 | Req根拠 |
|---|------|------|---------|
| 1 | Record CRUD | 作成/読取/更新/削除。統一構造 | Ch.4.3 |
| 2 | text Archetype | bodyの表示（プレーンテキスト+最小Markdown）| Ch.6.2 |
| 3 | Revision生成 | 更新時に新Revision作成。非破壊 | Ch.4.4 |
| 4 | Relation CRUD | structural + categorical の2種 | Ch.7 |
| 5 | フォルダ（structural） | Record型フォルダ + structural Relation | Ch.8.3 |
| 6 | タグ（categorical） | Record型タグ + categorical Relation | Ch.8.5 |
| 7 | 時間モデル | 全オブジェクトにcreated_at/updated_at | Ch.9 |
| 8 | 最小UI（3ペイン） | list + detail + meta の基本レイアウト | Ch.13 |
| 9 | IDB永続化 | IndexedDBへの保存/読込 | — |
| 10 | HTML Export（rehydratable） | 単一HTMLとしてExport | Ch.3.5 |
| 11 | HTML Rehydrate | ExportしたHTMLからデータ復元 | Ch.3.5 |

### 含めないもの
- todo Archetype（Phase 2）
- form Archetype（Phase 2以降）
- attachment/Asset実装（Phase 2）
- Force Graph / タイムラインビュー
- 逆算型進捗管理
- 活動量ダッシュボード
- テンプレート/クローン
- 検索・フィルタ（最小のtitleフィルタを除く）
- Gen1移行
- 高度Markdown（KaTeX等）
- スナップショット

### requirements対応
- Ch.4: Record/Revision/Relation の基本CRUD
- Ch.5-6: text Archetype のみ
- Ch.7: structural + categorical Relation
- Ch.8: フォルダとタグ（最小）
- Ch.9: 時間モデル
- Ch.3.5: 単一HTML

### 完了条件
1. テキストRecordを作成・編集・表示できる
2. フォルダを作成し、Recordをフォルダに入れられる
3. タグを作成し、Recordにタグ付けできる
4. 編集時にRevisionが生成され、履歴が見られる
5. 単一HTMLとしてExportし、そのHTMLを開いてデータが復元される
6. IDBに自動保存され、ブラウザ再起動後も復元される
7. 上記すべてに対するテストが存在する

### 想定リスク
- Relation正規化とUIの橋渡しに想定外の複雑さ
- rehydrate（HTML→データ復元）の実装難易度
- 「最小」の線引きが甘くなりスコープクリープ

## Phase 2: Practical Daily-Use MVP

### 目的
日常利用に耐える最小限の実用版を完成させる。
PKC1のユーザーが「乗り換えてもよい」と感じる最低ライン。

### 含めるもの

| # | 機能 | 詳細 | Req根拠 |
|---|------|------|---------|
| 1 | todo Archetype | title + due_date + status(open/done) | Ch.6.4 |
| 2 | attachment Archetype | Asset管理、ファイル添付 | Ch.6.6 |
| 3 | semantic Relation | 参照リンク（Record間の意味的関連） | Ch.7.4.3 |
| 4 | 検索・フィルタ | title/body全文、type別、tag別 | — |
| 5 | タイムラインビュー | グローバルタイムライン（最小版） | Ch.10.3 |
| 6 | レコード履歴表示 | リビジョン一覧、差分表示 | Ch.10.4 |
| 7 | ローカルファイル同期 | File System Access API | — |
| 8 | JSON Export/Import | PKC JSON形式での入出力 | — |
| 9 | ドラッグ&ドロップ | ファイル/テキストの取り込み | — |
| 10 | Markdown強化 | marked.js連携、コードハイライト | — |
| 11 | textlog Archetype | 時刻付き追記型テキスト | Ch.6.3 |
| 12 | 基本操作シナリオ文書 | ユーザー利用手順の文書化 | — |

### 含めないもの
- form Archetype（feature moduleとして別途）
- 逆算型進捗管理
- 活動量ダッシュボード
- テンプレート/クローン
- Gen1移行ツール
- EMLスレッド
- Force Graph（feature moduleとして別途）

### requirements対応
- Ch.6: text + textlog + todo + attachment
- Ch.7: structural + categorical + semantic
- Ch.10: グローバルタイムライン + レコード履歴
- Ch.3.5: 単一HTML（完全版）

### 完了条件
1. Phase 1の完了条件すべて
2. TODOレコードを作成し、期日設定・完了操作ができる
3. ファイルを添付でき、画像はインライン表示される
4. レコード間リンクを作成でき、リンク先に遷移できる
5. タイムラインビューで時系列一覧が表示される
6. レコードの変更履歴を閲覧でき、差分が確認できる
7. ファイルにローカル保存され、自動同期される
8. JSON形式でのExport/Importが動作する
9. 基本操作シナリオ文書が完成している

### 想定リスク
- Asset管理の容量問題（base64の肥大化）
- タイムラインビューのパフォーマンス
- ローカルファイル同期のブラウザ互換性
- PKC1ユーザーの期待値とのギャップ
