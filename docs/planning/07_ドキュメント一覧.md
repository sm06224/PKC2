## 7.2 操作手順系ドキュメントの重要性

### なぜ操作手順文書を重視するか

PKC1で発生した問題の多くは**操作順序依存**に起因する:

- IIFE後付けパッチがDOMの状態に依存し、初期化順序で挙動が変わる
- グローバル状態Sの書き換えタイミングによってUI不整合が発生
- フォーム保存→フォルダ作成→添付ファイル登録 の順序が暗黙に決まっている
- スナップショット取得とエントリ編集の競合

### 操作シナリオ文書の構成案

```
docs/scenarios/
├── SC-01_レコード作成と編集.md
├── SC-02_フォルダ整理.md
├── SC-03_タグ付けと検索.md
├── SC-04_リビジョン確認と差分.md
├── SC-05_HTMLエクスポートと再読込.md
├── SC-06_TODO作成と完了.md
├── SC-07_ファイル添付.md
├── SC-08_レコード間リンク.md
└── SC-09_ローカルファイル同期.md
```

各シナリオは以下の形式で記述:

```markdown
# SC-XX: シナリオ名

## 前提条件
- （例: PKCが1つ以上のRecordを持つ状態）

## 操作手順
1. ユーザーが○○する
2. システムが○○を表示する
3. ユーザーが○○する
4. システムが○○を保存する

## 期待結果
- ○○が表示される
- ○○がIDBに保存される

## ���作順序の注意点
- ⚠ ステップ2の前にステップ3を行った場合: ○○となる
- ⚠ ステップ4中にブラウザを閉じた場合: ○○となる

## 関連テスト
- test/scenarios/sc-xx.test.ts
```

## 7.3 UXフロー文書の構成案

ux-flows.md は操作シナリオより抽象度が高く、
画面遷移とユーザーの意思決定ポイントを記述する。

```
[起動] → [コンテナ読込] → [メイン画面]
                                 │
                    ┌────────────┼────────────┐
                    ▼            ▼            ▼
              [Record作成]  [Record選択]  [Export]
                    │            │            │
                    ▼            ▼            ▼
              [編集・保存]  [詳細表示]    [HTML出力]
                                 │
                    ┌────────────┼────────────┐
                    ▼            ▼            ▼
              [履歴表示]  [Relation追加]  [タグ追加]
```

各ノードで「何が表示されているか」「何が操作可能か」を明記し、
操作順序依存が発生しうるポイントにマーカーを付ける。
