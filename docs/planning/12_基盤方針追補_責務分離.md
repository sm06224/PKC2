# 12. 基盤方針追補 — 真の単一HTMLアプリに必要な責務分離

本文書は `09_ディレクトリ構成.md` を再構成し、
core / adapter / feature / embed runtime / release builder の5層を定義する。

---

## 12.1 5層アーキテクチャ

```
┌────────────────────────────────────────────┐
│  Layer 5: release-builder                   │
│  単一HTML生成、metadata埋め込み、CI連携       │
│  ※ ビルド時のみ実行、ランタイムには含まれない   │
└────────────────────────────────────────────┘
┌────────────────────────────────────────────┐
│  Layer 4: embed-runtime                     │
│  rehydrate, clone, export, sandbox,         │
│  PKC-Message, IDB永続化                      │
│  ※ ブラウザ内でのみ実行                       │
└────────────────────────────────────────────┘
┌────────────────────────────────────────────┐
│  Layer 3: feature modules                   │
│  search, markdown, form, graph, dashboard   │
│  ※ core + adapter に依存。feature間は疎結合   │
└────────────────────────────────────────────┘
┌────────────────────────────────────────────┐
│  Layer 2: adapter                           │
│  外部ライブラリラッパー、ブラウザAPI抽象化      │
│  UI shell, state management                 │
│  ※ core に依存。ブラウザAPIに依存してよい       │
└────────────────────────────────────────────┘
┌────────────────────────────────────────────┐
│  Layer 1: core                              │
│  model, archetype, operations               │
│  ※ 一切の外部依存なし。ブラウザAPI非依存       │
│  ※ Node.js / Deno / ブラウザで等しくテスト可能 │
└────────────────────────────────────────────┘
```

**依存方向: Layer N は Layer 1〜(N-1) にのみ依存できる。逆方向の依存は禁止。**

---

## 12.2 各層の詳細

### Layer 1: core（汚染禁止ゾーン）

```
src/core/
├── model/           # Record, Relation, Revision, Asset, Container 型
├── archetype/       # Archetype インターフェース + 組込み類型
├── operations/      # CRUD, query, validation — 純関数のみ
├── id.ts            # LID/RID 生成（crypto.randomUUID のみ許容）
└── time.ts          # 時刻生成（Date.now / ISO変換のみ許容）
```

**規約**:
- `import` できるのは core 内の他モジュールのみ
- DOM API 禁止（document, window, navigator, localStorage, IndexedDB 等）
- Node.js API 禁止（fs, path, process 等）
- 外部 npm パッケージ禁止
- 例外: `crypto.randomUUID()`（ブラウザ/Node 両方で利用可能）

**テスト**: `vitest` で Node.js 環境で実行可能であること

### Layer 2: adapter（橋渡し層）

```
src/adapter/
├── state/           # 状態管理（Store, 状態機械, action dispatch）
├── ui/              # UI shell, components, views
│   ├── shell/
│   ├── views/
│   ├── components/
│   └── styles/
├── platform/        # ブラウザAPI抽象化
│   ├── dom.ts       # DOM操作ヘルパー
│   ├── clipboard.ts
│   └── file-access.ts
└── external/        # 外部ライブラリラッパー
    ├── markdown-adapter.ts
    └── diff-adapter.ts
```

**規約**:
- core に依存してよい
- ブラウザ API に依存してよい（ただし `platform/` 経由で抽象化）
- 外部ライブラリは `external/` 経由のみ。直接 import 禁止
- feature modules に依存してはならない

### Layer 3: feature modules（拡張機能群）

```
src/features/
├── search/          # 検索・フィルタ・ソート
├── relation/        # selector, tag-selector, tag-filter（categorical → tags → filter）
├── markdown/        # 高度Markdown（KaTeX, hljs等）
├── form/            # FORM Archetype
├── graph/           # Force Graph
├── dashboard/       # 活動量ダッシュボード
└── textlog/         # TextLog Archetype 拡張
```

**規約**:
- core + adapter に依存してよい
- feature 間の依存は禁止（search が form に依存しない等）
- 各 feature は独立して有効/無効化できる設計
- feature の登録は `src/main.ts` のブートストラップで行う

**実装パターン（Issue #20-#26 で確立）**:

feature の追加は以下の手順に従う:

1. `src/features/{name}/` に pure function を配置（core 型のみ import）
2. `src/features/index.ts` から re-export
3. feature state を `AppState` に runtime-only フィールドとして直置き
4. feature 用 UserAction を追加（ready phase でのみ処理、他 phase では block）
5. renderer が feature 関数を import し、state → DOM 投影に利用
6. action-binder が DOM event → feature action dispatch を担当

**feature state に関する判断**:
- feature state は AppState にフラット配置（namespace 未導入）
- 2〜3 feature 追加後に `AppState.features: FeatureState` への構造整理を検討
- result count 等の derived value は AppState に持たず renderer で算出
- 複数フィールドの一括リセットは専用 action で処理（`CLEAR_FILTERS` 等）
- feature 固有の型（SortKey, SortDirection）は feature module 内で定義（core/model に置かない）
- UserAction で feature 型を参照する場合は inline string literal union を使用（core→feature 依存回避）
- 操作パイプライン順序: query → archetype → tag → sort（CLEAR_FILTERS は filter 3種をリセット、sort は独立）
- detail view の body 表示は DetailPresenter で archetype dispatch（shared chrome は renderer）
- registerPresenter() で archetype 別 presenter 追加可能（未登録時は default text presenter）

### Layer 4: embed-runtime（ブラウザ実行時専用）

```
src/runtime/
├── rehydrate.ts     # HTML → データ復元
├── export.ts        # データ → HTML 埋め込み export
├── clone.ts         # コンテナ複製
├── sandbox.ts       # iframe 埋め込み制御
├── message.ts       # PKC-Message 送受信
└── idb.ts           # IndexedDB 永続化
```

**規約**:
- core + adapter に依存してよい
- ブラウザ API に強く依存する（DOM解析、Blob、IDB 等）
- 配布HTMLに含まれるコード

### Layer 5: release-builder（ビルド時専用）

```
build/
├── vite.config.ts       # Stage 1: Vite 設定
├── release-builder.ts   # Stage 2: 単一HTML組み立て
├── shell.html           # HTMLテンプレート
├── version.ts           # バージョン情報生成
└── integrity.ts         # integrity hash 計算
```

**規約**:
- Node.js で実行。ブラウザでは実行されない
- 配布HTMLには含まれない
- src/ のコードを import しない（ビルド出力のみ利用）

## 12.4 CSS の動的差し替え方針

requirements の「同一データを複数ビューで投影」を CSS レベルでも保証するため:

1. **CSS 変数によるテーマ制御**: `:root` に CSS custom properties を定義
2. **レイアウトCSS と テーマCSS の分離**: `base.css`（構造）と `theme.css`（色・フォント）
3. **ランタイム差し替え**: `<style id="pkc-theme">` を書き換えることでテーマ変更
4. **clone 時のテーマ継承**: clone 元のテーマ設定をデータとして保持し、clone 先に適用

```css
/* base.css — 構造。minifyされても問題ない */
#pkc-root { display: grid; grid-template-columns: var(--sidebar-w) 1fr var(--meta-w); }

/* theme.css — 差し替え可能 */
:root {
  --sidebar-w: 260px;
  --meta-w: 280px;
  --c-bg: #ffffff;
  --c-text: #1a1a1a;
  --c-accent: #2563eb;
}
```
