# 日常操作

エントリの作成から削除・復元、関連付けまでの基本操作を網羅します。

## エントリの作成

ツールバーの **+ Note** / **+ Todo** / **+ Form** / **+ File** のいずれかを押します。

- **+ Note** — text archetype の新規エントリ
- **+ Todo** — todo archetype の新規エントリ
- **+ Form** — form archetype の新規エントリ
- **+ File** — ファイル選択ダイアログ → attachment archetype

作成直後は編集モードに入り、すぐにタイトルと本文を入力できます。

## 編集サイクル

PKC2 の編集は次のフェーズで進みます。

1. **閲覧（ready）** — サイドバーでエントリを選択すると詳細ペインに表示
2. **編集開始（editing）** — 編集ボタン、またはダブルクリックで編集モードへ
3. **編集確定（commit）** — 保存ボタン、または `Ctrl+S` / `Cmd+S`
4. **編集キャンセル（cancel）** — Escape キー、またはキャンセルボタン

編集中は他のエントリに移動できません（編集中の変更を誤って失わないためのガードです）。

## クイック更新

いくつかの操作は編集モードに入らずに直接実行できます。

- **Todo のステータス切替** — Todo 上のチェックボックス `[ ]` / `[x]` をクリック
- **TextLog の追記** — 編集モードで `Ctrl+Enter` で確定する、または追記欄から直接追記
- **TextLog の★フラグ切替** — 追記単位の星アイコンをクリック
- **HTML 添付の sandbox 許可 checkbox** — 情報パネル側の「Sandbox Policy」で個別にトグル

これらのクイック更新は本文だけを書き換え、編集フェーズには遷移しません。クリック直後にペイン全体が再描画されますが、**スクロール位置はそのまま維持**されるので、長い TEXTLOG の下の方のログや、attachment の奥の項目を触っても視界がリセットされません。

## 編集中の入力補助

編集中に使える小さな入力補助が 2 つあります。それぞれ対象範囲が異なります（詳細は各節を参照）。タイトル欄・form では動作しません。

### リンク付きコピーの貼り付け

Web ブラウザ等からリンクをコピーして貼り付けると、自動で Markdown リンクに正規化されます。

- **対象**: TEXT エントリの本文 textarea / TEXTLOG の追記欄（textlog-append-text）/ TEXTLOG のログ入力欄（textlog-entry-text）
- **非対象**: タイトル欄・form など
- **条件**: クリップボードに text/html が含まれていて、かつリンク（`<a href>`）が 1 つ以上あること
  - プレーンテキストだけのコピーでは従来通り、そのままの文字列として貼り付けられます
- **出力形式**:
  - ラベルと URL が異なる場合（例: 「Example」というラベルで `https://example.com` をコピー）→ `[Example](https://example.com)` 形式のリンク
  - ラベルと URL が同じ（アドレスバーから URL をそのままコピーした場合など）→ `https://example.com` だけの bare URL（`[URL](URL)` の冗長形は作りません）
- **安全**: `javascript:` / `data:` などの危険な URL はリンク化されず、ラベルのみ通常テキストとして残ります

### bare URL の表示時リンク化（fallback 表示）

上の「リンク付きコピーの貼り付け」は貼り付け時にクリップボードを整形する経路です。これとは別に、PKC2 は **表示時** にも bare URL を自動でクリック可能なリンクとして描画します。

- **発火条件**: 本文に `https://` / `http://` / `ftp://` で始まる URL がそのまま書かれていること
- **挙動**: read（閲覧）表示のとき、該当する URL 部分だけが `<a>` に変換され、`target="_blank"` + `rel="noopener noreferrer"` 付きで開ける状態になります
- **保存データ**: 本文（保存される `body`）は **bare URL のまま**。リンク化は表示レンダリング時だけの処理で、保存内容には一切書き込まれません
- **編集中 textarea**: 編集モードの textarea では **plain text（URL 文字列そのもの）** のまま見えます。`<a>` タグがテキスト欄に混ざることはありません
- **安全**: `javascript:` / `file://` / `data:text/html` などの危険なスキームはリンク化されず、そのままのテキストとして表示されます

この挙動は、貼り付け時にクリップボードへ HTML anchor が渡されないケース（例: Firefox の macOS でアドレスバーから URL をコピー）で実用になります。貼り付け経路が何もしなくても、結果的に表示時にはクリックできます。

### 本文の検索・置換（Find & Replace）

編集中の TEXT 本文に対して、一括置換ダイアログを開けます。

- **開き方**: 編集モードの action bar にある `🔎 Replace` ボタン（TEXT エントリを編集中のときにだけ表示されます）
- **入力**:
  - **Find** — 検索文字列
  - **Replace with** — 置換文字列
- **オプション**
  - **Regex** — JavaScript の正規表現として Find を解釈。`$1` などの back-reference が使えます
  - **Case sensitive** — 大文字小文字を区別
  - **Selection only** — ダイアログを開く **前** に本文 textarea で文字列を選択しておくと、検索・置換の対象が選択範囲内に限定されます。選択せずに開いた場合はこのオプションは灰色になり使えません。選び直したい場合はいったんダイアログを閉じて、選択し直してから再度開いてください
- **プレビュー**: 状態行に「N match(es) will be replaced.」とヒット件数がリアルタイムに表示されます
- **Apply**: ヒット件数が 1 以上のときだけ有効。押すと一括で置換され、件数が 0 に更新されます（ダイアログは開いたまま、続けて別の置換ができます）
- **閉じ方**: Close ボタン / ダイアログ外側クリック / `Escape`。閉じた後は本文 textarea にフォーカスが戻ります
- **Enter キー**: Find / Replace 入力欄で Enter を押すと、Apply が有効なときは即時実行されます

#### Apply が無効になる条件

- Find 欄が空
- Regex が有効で、正規表現として無効な文字列（状態行に赤文字でエラーメッセージが表示されます）
- ヒット 0 件（`No matches in current entry.`）

#### 対象外

- タイトル / source_url / tags の置換
- form / attachment / folder の本文
- **textlog の本文**（この TEXT Replace ダイアログは TEXT 専用です。textlog には **ログ行ごと** の別ダイアログがあります — 後述の「TEXTLOG のログ行検索・置換」を参照）
- **readonly** ワークスペース（そもそも編集モードに入れないため `🔎 Replace` ボタンは出ません）
- **historical revision** を閲覧中（編集モードに入れないため同上）
- 全エントリ一括置換・複数エントリ置換

## TEXTLOG のログ行検索・置換

編集中の TEXTLOG で、**特定のログ行 1 件** の本文だけを対象にした Find & Replace ダイアログを開けます。TEXT の Find & Replace とは別経路です（ログ単位で独立）。

### 開き方

TEXTLOG を編集モードで開くと、各ログ行に `🔎` ボタンが表示されます（ログ 1 件につき 1 つずつ）。そのボタンを押すとダイアログが開き、**そのログ行の本文だけ** を対象にします。

- ダイアログを開いた時点で対象ログが決まります。後から別のログを対象にしたい場合は、いったん Close → 別の行の `🔎` を押してください
- タイトル: `Find & Replace (current log)`
- 対象表示: 状態行に `N matches will be replaced in current log.` と表示されます

### 入力とオプション

- **Find** — 検索文字列
- **Replace with** — 置換文字列
- **Regex** — JavaScript の正規表現として Find を解釈。`$1` などの back-reference は Regex ON のときだけ使えます
- **Case sensitive** — 大文字小文字を区別

v1 では **Selection only は非対応** です（TEXT 側にはあります）。ログ行全体が対象になります。

### Apply の効果と対象範囲

- **変わるもの**: 起動したログ行 1 件の **本文** のみ
- **変わらないもの**（意図的な仕様）:
  - そのログの **タイムスタンプ**
  - そのログの **★フラグ**
  - そのログの **ログ ID**
  - 他のログ行の本文 / タイムスタンプ / フラグ / ID
  - ログの並び順（保存順は常に古→新の昇順に正規化されます）
  - **追記欄**（Ctrl+Enter で新規追記する欄）の未送信下書き
  - viewer の表示順（desc / asc）設定

### 対象外

- **TEXTLOG 全体の一括置換**（whole textlog）は v1 では提供しません
- **選択した複数ログの同時置換**（selected lines）も v1 では非対応
- **追記欄の下書き** はダイアログの対象外
- **viewer（閲覧モード）** では `🔎` ボタンは表示されません。編集モードに入る必要があります
- **readonly ワークスペース** / **historical revision 閲覧** / **preservation viewer** では編集モードに入れないため `🔎` ボタンは表示されません
- **複数エントリ / 全 container の一括置換** は提供しません

### Enter / Close / Escape

- Find / Replace 入力欄で Enter を押すと、Apply が有効なら即実行されます
- Close ボタン / ダイアログ外側クリック / `Escape` で閉じます
- 閉じた後は対象だったログ本文の textarea にフォーカスが戻ります

## TEXTLOG への画像貼り付け

TEXTLOG を編集モードで開いているとき、各ログセルのテキストエリアに画像をペーストできます（スクリーンショットや画像ファイルのコピー&ペースト）。

- **ペースト先**: フォーカスしているログセルのみ。複数ログが存在する場合でも、**現在フォーカスしているセルにだけ** アセットリンク `![名前](asset:...)` が挿入されます
- **複数ログ時の動作**: 2 件目以降のログセルにフォーカスしてペーストしても、先頭ログではなく、そのセルに正しく挿入されます
- **フォーカスの維持**: ペースト後もフォーカスと cursor はペーストしたセルの挿入点直後に留まります

リンク貼り付け（Markdown リンク自動整形）とは別の経路です。詳しくは「リンク付きコピーの貼り付け」を参照してください。

## 画像の自動最適化（Image Intake Optimization）

大きな画像（512 KB 以上の JPEG・PNG・WebP・BMP）を貼り付け・ドロップ・ボタン添付すると、PKC2 は保存前に最適化を提案します。最適化は WebP 形式への変換と、長辺 2560 px への縮小を組み合わせたもので、ストレージ容量を大幅に削減できます。

### 最適化が提案される条件

以下のすべてを満たす画像が対象です。

- MIME が `image/jpeg`・`image/png`・`image/webp`・`image/bmp` のいずれか
- ファイルサイズが 512 KB 以上
- 透過チャンネルを持たない（透過 PNG はスキップされます）
- Canvas による変換で実際にサイズが小さくなる

GIF・SVG やその他の非画像ファイルは最適化対象外で、そのまま保存されます。

### 確認ダイアログ

条件を満たすと確認ダイアログが表示されます。

| ボタン | 動作 |
|--------|------|
| **最適化して保存** | WebP に変換して保存（デフォルト） |
| **元のまま保存** | 最適化せず原画のまま保存 |

**原画も保持する** チェックボックスを ON にすると、最適化後の画像に加えて原画も内部に保存されます（デュアル保存）。表示や通常利用には最適化後の画像が使われますが、原画はエクスポート時に含まれます。

### 記憶設定（Remembered Choice）

確認ダイアログの **次回から確認しない** チェックを ON にして決定すると、同じ操作経路（下記）では以降の確認を省略して自動適用されます。

| 操作経路 | 記憶キー |
|---------|---------|
| 貼り付け（Ctrl+V / Cmd+V） | paste |
| ドラッグ&ドロップ | drop |
| 📎 File ボタン | attach |

記憶は操作経路ごとに独立しています。たとえば「貼り付けでは常に最適化」と記憶しても、ドロップやボタン添付には影響しません。

記憶設定を解除するには、ブラウザの localStorage から該当キー（`pkc2.imageOptimize.preference.paste` 等）を手動で削除してください。

### 最適化がスキップされるケース

以下の場合は確認ダイアログを出さず、元のファイルがそのまま保存されます。

- GIF・SVG など非対応形式
- 512 KB 未満の小さな画像
- 透過チャンネルを持つ PNG
- Canvas による変換が失敗した場合
- 変換後のサイズが元より大きくなる場合

スキップ時はトースト通知で理由が表示されます（透過 PNG / 変換失敗 / サイズ変わらず）。

## TEXTLOG の画像の段階的表示（Staged Render）

画像を多く含む TEXTLOG を表示するとき、PKC2 はログ行（log article）を **段階的に描画** することで初期表示を軽くしています。対象は TEXTLOG の **閲覧モード** のみで、TEXT / Todo / FOLDER 等 他の archetype には影響しません。

### 初期表示の動作

TEXTLOG を開いた直後は、上から順に **先頭 8 件のログ行だけ** が画像付きで即時描画されます。9 件目以降は **placeholder（見出しのみの空白カード）** として表示され、画像や本文テキストの描画は保留されます。

- placeholder にもタイムスタンプ・重要フラグ・行リンクボタンは表示されます — 「そこにログが存在する」こと自体は見失いません
- placeholder の高さは最低 160 px 確保されており、スクロール位置が大きくずれることはありません
- ログ行が 8 件以下の TEXTLOG では placeholder は発生せず、従来どおり全件が即時描画されます

### スクロールによる追加描画（Hydrate）

画面を下にスクロールして placeholder が viewport に **近づく**（viewport 下端まで約 400 px）と、その placeholder が自動的に本物のログ行に置き換わり、画像・本文テキストが描画されます。この置き換えを **hydrate**（水和）と呼びます。

- hydrate 済みのログ行は 2 度目以降のスクロールでも再描画されません
- スクロール開始直後には先頭 8 件に続いて **4 件** が先読み hydrate され、体感上は「画像が次々に展開していく」印象になります
- 読み込み完了までに時間が掛かる画像でも、placeholder 自体は即座に消え、ログ行が実体化されます（描画の順序：hydrate → 画像デコード → 表示）

### 印刷時の一括展開

印刷ダイアログを開く（`Ctrl+P` / `Cmd+P` / ブラウザの印刷メニュー）と、ブラウザの `beforeprint` イベントで **残っている placeholder をすべて即時 hydrate** します。印刷プレビューやエクスポートされた PDF に placeholder のままのログが含まれることはありません。

### 選択モードとの整合

**TEXTLOG → TEXT 変換** のためのログ選択モードに入ると、placeholder にも実ログと同じ **選択チェックボックス** が表示されます。選択したまま下にスクロールしてログを hydrate しても、チェック状態は維持されます（状態は AppState 側に保持されるため、DOM 置き換えを経ても失われません）。

### 編集モードとの関係

編集モード（`edit`）に入ったときは staged render は適用されず、全ログ行がすぐに編集可能な形で描画されます。編集を終えて閲覧モードに戻ると、再び staged render が適用されます（冒頭 8 件が描画、以降は placeholder から再開）。

## ファイルの添付（通常操作 / ready フェーズ）

エントリを**編集モードで開いていないとき**（ready フェーズ）にファイルを添付する方法は 2 種類あります。

### センターペインへドロップ

センターペイン中央のドロップゾーン（「Drop files here」と表示されているエリア）にファイルをドロップすると、attachment エントリが作成されます。

### サイドバー常設ドロップゾーン

サイドバー最下部にも常設のドロップゾーン（「📎 Drop files here」）があります。エントリ一覧の下に常時表示されており、センターペインを表示していなくても使えます。

### 複数ファイルを一度にドロップ

複数のファイルをまとめて選んでドロップすると、**ドロップした FileList の順序**（ファイル選択/DnD の内部順）で 1 件ずつ順番に attachment エントリが作成されます。並列処理は行われないため、ファイルの追加順が予測可能です。

- 途中のファイルが読み込みエラーになっても、**残りのファイルの処理は続きます**
- ファイルサイズ警告が出ても同様に処理は継続します

### 重複内容のファイルをドロップしたとき

同一のファイル（ハッシュ値とファイルサイズの両方が既存の添付と一致する場合）を再度ドロップすると、「`「ファイル名」は既存の添付と同一内容です`」という情報トーストが表示されます。

- トーストはあくまで **情報通知** です。添付エントリの作成は必ず行われます（スキップされません）
- 意図的に同名ファイルを複数添付したい場合もそのまま実行できます

### editing フェーズ中の挙動

エントリを**編集モードで開いているとき（editing フェーズ）**は、サイドバー常設ドロップゾーンは非アクティブ（薄く表示、クリック不可）になります。editing フェーズ中のファイル添付は、センターペインの編集エリアへのドロップ（下記「編集中のドラッグ&ドロップ / ボタン添付」）が担います。

## 編集中のドラッグ&ドロップ / ボタン添付（自動リンク挿入）

TEXT / TEXTLOG / FOLDER のいずれかのエントリを**編集モードで開いているとき**に、センターペインへファイルをドラッグ&ドロップするか、**📎 File** ボタンを押してファイルを選ぶと、次の 2 つが同時に行われます。

1. attachment エントリが作成される（既存の添付動作と同じ）
2. 操作開始時点でフォーカスしていた本文 textarea のカーソル位置に内部リンクが挿入される

### 挿入されるリンク形式

| ファイル種別 | 挿入形式 |
|------------|---------|
| 画像（MIME が `image/*`） | `![ファイル名](asset:キー)` |
| 非画像（PDF・テキストなど） | `[ファイル名](asset:キー)` |

### 対象 archetype と挿入先

| Archetype | 挿入先 textarea |
|-----------|----------------|
| **text** | 本文（body） |
| **folder** | 本文（body） |
| **textlog**（ログセルにフォーカス中） | フォーカスしているログセルのみ |
| **textlog**（追記欄にフォーカス中） | 追記欄（textlog-append-text） |

### TEXTLOG の挿入先

TEXTLOG を編集中の場合、**操作開始時点でフォーカスしているログセルにだけ**リンクが挿入されます。他のログセルは変更されません。どのセルにフォーカスが当たっているか確認したい場合は、ドロップ前にそのセルをクリックしてください。

### 複数ファイルを一度にドロップ / 選択した場合

ファイルの数だけ attachment エントリが作成され、各ファイルのリンクが `\n` で区切られてカーソル位置に順番に挿入されます。

```
![img1.png](asset:ast-aaa)
[doc.pdf](asset:ast-bbb)
```

### リンクが挿入されない条件

以下の場合は attachment エントリは作成されますが、本文へのリンク挿入は行われません。

- 編集モードに入っていない（ready / 閲覧モード）
- 編集中だが本文 textarea にフォーカスが当たっていない（TEXTLOG で複数セルがある場合、どのセルも選ばずにドロップした場合など）
- todo / form / attachment archetype を編集中（これらは対象外）

### スクリーンショット貼り付けとの関係

クリップボードからの画像ペースト（スクリーンショット）は以前から自動リンク挿入に対応しています。DnD / ボタン添付もこれと同じ挿入動作を行います。添付の経路を問わず、本文への参照リンクが自動で挿入されます。

## 削除と復元

### 削除

1. 削除したいエントリを選択
2. 削除ボタン、または複数選択して一括削除
3. 確認ダイアログで **OK**

削除されたエントリはリビジョン履歴から復元できます（完全消去ではありません）。

### 復元

1. リビジョン履歴を開く
2. 復元したい時点のスナップショットを選択
3. 復元ボタン

復元は「元の状態に戻す」ではなく「スナップショットの内容で今を上書きする」動作です。復元前の状態も別のリビジョンとして自動で記録されるので、復元を取り消したくなっても履歴は失われません。

**制約**: リビジョンに記録された archetype と現在の archetype が違う場合は、復元は拒否されます（TEXT エントリに TODO の JSON を書き戻してしまうような事故を防ぐためのガードです）。

### ゴミ箱の完全消去

リビジョン履歴も含めて本当に消したい場合は **Purge Trash** を使います。これは取り消せない操作です。

## 複数選択と一括操作（Multi-select）

サイドバー・Calendar・Kanban のどのビューからでも、複数のエントリをまとめて操作できます。

### 選択方法

| 操作 | 意味 |
|------|------|
| クリック | そのエントリだけを選択（他は解除） |
| `Ctrl+Click` / `Cmd+Click` | 個別に追加 / 解除 |
| `Shift+Click` | 直前の選択から範囲選択 |
| `Escape` | 複数選択をクリア |

選択中のエントリには強調表示が付きます。選択数はツールバーの一括操作ボタンの有効・無効で確認できます。

### 使える一括操作

- **一括削除** — 選択中のエントリをまとめて削除
- **フォルダへ一括移動** — 選択中のエントリを指定フォルダの子にする
- **ルートへ一括移動** — 選択中のエントリをフォルダ配下から外す
- **一括ステータス切替**（Kanban / Calendar でのみ）— 選択中の Todo を `open` / `done` に揃える
- **一括期日設定 / クリア**（Kanban / Calendar でのみ）— 選択中の Todo の期日を同時に書き換え

### 注意点

- ビューを切り替えても選択は維持されますが、対象のエントリが新しいビューに表示されない場合は操作ボタンが無効になります
- フィルタや並び順を変えると、視覚的な並びと選択の順序が一致しなくなることがあります。一括削除の前に対象件数を確認してください

## サイドバーの並び替え（Manual Order）

サイドバーの並び順は通常、`Title` / `Created` / `Updated` のいずれかで自動ソートされます。**Manual** モードに切り替えると、エントリを 1 件ずつ上下に動かして任意の順序に並べ替えられます。

### Manual モードへの切り替え

- サイドバー上部の並び順セレクタで **Manual** を選びます
- Manual を選んだ瞬間、現在のサイドバーの並びがそのまま手動順として確定します
- 他のソート（Title / Created / Updated）に戻すと手動順は画面上では使われなくなりますが、Container には保持されたままなので、再度 Manual に戻すと同じ並びが復元されます

### 上下への 1 件移動（Move up / Move down）

- サイドバーでエントリを選択すると、その行に **↑（上へ）** と **↓（下へ）** のボタンが現れます
- **↑** を押すと 1 つ上の同グループのエントリと位置が入れ替わります
- **↓** を押すと 1 つ下の同グループのエントリと位置が入れ替わります
- 端（先頭 / 末尾）で対応するボタンを押しても何も起こりません（エラーにはなりません）

### どの範囲で動くか

| エントリの所属 | 入れ替え対象 |
| --- | --- |
| ルート直下のエントリ | 同じくルート直下の前後エントリ |
| フォルダの子エントリ | 同じフォルダの子エントリのうち前後のもの |
| フォルダ自身 | 兄弟フォルダ / 同階層の通常エントリ |

親子を飛び越える移動（フォルダ外へ出す / 別フォルダへ移す）は Manual Order では扱いません。ツリーの構造を変えたい場合は従来どおりドラッグ&ドロップを使ってください。

### 並び替えボタンが表示される条件

以下をすべて満たしたときだけ ↑ / ↓ ボタンが出ます。

- 並び順セレクタが **Manual** である
- 現在のビューが **Detail**（Calendar / Kanban では表示されません）
- エントリが **選択されている**
- 閲覧専用（read-only）モードではない
- **Import プレビュー** を開いていない

このいずれかを満たさない場合、ボタン自体が出ません（無効化ではなく非表示です）。

### フィルタ / 検索中の挙動

- フィルタや検索でサイドバーに一部のエントリだけが出ているときも、Manual 並びは **Container 全体** に対して保持されます
- 画面上で隣り合って見えても、実際には間に非表示のエントリが挟まっている場合があります。そのため **↑ / ↓ は画面の隣ではなく、Container 上の直前 / 直後との入れ替え** として働きます
- フィルタを外したときに「見た目が変わった」と感じても、手動順そのものは壊れていません

### v1 での非対象

以下は現時点の Manual Order では扱いません。

- サイドバーでのドラッグ&ドロップによる並び替え
- 複数選択の一括並び替え
- 「先頭へ」「末尾へ」などのジャンプ操作
- Calendar / Kanban / TextLog ビューでの並び替え
- archive 済みエントリの順序操作

これらは通常の編集・削除・DnD・フィルタ機能でカバーできる範囲です。

## サイドバーの種別フィルタ（Archetype Filter）

サイドバー上部の種別フィルタバーで、表示するエントリを archetype（種別）で絞り込めます。複数の種別を同時に選べます。

### 基本操作

| 操作 | 効果 |
|------|------|
| 個別ボタン（Text / Textlog / Folder など）を押す | その種別のON/OFFを切り替え |
| **All** ボタンを押す | 選択をすべて解除して全エントリを表示 |
| **×** クリア（検索欄横） | 種別フィルタ・テキスト検索・タグフィルタをすべてリセット |

### 全解除 = 全表示

種別を 1 つも選んでいない状態（初期状態）は **全表示** です。「何も選ばない → 何も出ない」ではありません。1 種別だけ選ぶと従来の単一フィルタと同じ結果になり、2 種別以上選ぶとそれら **すべてに該当するエントリが合わせて表示** されます（OR 結合）。

### Primary / Secondary の 2 段構成

フィルタバーは頻度の高い種別と低い種別で 2 段に分かれています。

| 段 | 含まれる種別 | 既定表示 |
|----|------------|---------|
| Primary | Text / Textlog / Folder | 常時表示 |
| Secondary | Todo / File / Form / Generic / Opaque | 折りたたみ（▼ で展開） |

展開ボタン（▼ / ▲）を押すと Secondary 段が現れます。展開状態はフィルタを × でクリアしても維持されます。

### テキスト検索・タグとの組み合わせ

種別フィルタは、テキスト検索（searchQuery）およびタグフィルタと **AND 結合** で併用できます。たとえば種別「Text」＋テキスト検索「メモ」で、Text エントリのうち「メモ」を含むものだけを絞り込めます。

### v1 での非対象

- フィルタ選択の永続化（現在はセッション限り）
- 各ボタンに一致件数を表示
- キーボードショートカットでの種別切り替え
- フィルタプリセットの保存

## 外観と地域の設定（☰ Menu）

画面右上の **☰ Menu** から外観・フォント・言語・タイムゾーンをカスタマイズできます。**すべて保存されます**（ページをリロードしても維持されます）。保存先はコンテナ内の隠しエントリ `__settings__`（System Settings Entry）で、ワークスペースと一緒に持ち運べます。

| 設定軸 | UI | 保存 | 初期値 |
|---|---|---|---|
| **テーマ**（配色） | Theme 行（3 ボタン） | **あり** | System |
| **Scanline** | Scanline 行（2 ボタン） | **あり** | Off |
| **Accent color** | Accent 行（カラーピッカー + reset） | **あり** | Neon Green |
| **Border color** | Border 行（カラーピッカー + reset） | **あり** | System Default |
| **Background color** | Background 行（カラーピッカー + reset） | **あり** | System Default |
| **UI Text color** | UI Text 行（カラーピッカー + reset） | **あり** | System Default |
| **Body Text color** | Body Text 行（カラーピッカー + reset） | **あり** | System Default |
| **Font** | Font 行（プルダウン + 直接入力欄） | **あり** | System Default |
| **Language** | Language 行（プルダウン） | **あり** | System |
| **Timezone** | Timezone 行（プルダウン） | **あり** | System |
| **WCAG Contrast** | Contrast 行（表示のみ） | — | — |

すべての設定項目は **独立** しており、どれを変えても他の項目に影響しません。

Menu は画面の高さに応じて **スクロール** し、画面幅が 720px 以上の場合は **2 列レイアウト** に切り替わります。設定項目が多くてもすべて操作可能です。

Menu の **Contrast** 行には、現在の背景色と UI テキスト色・本文テキスト色の WCAG コントラスト比が **AA / AAA / Fail** のバッジ付きで表示されます。色を変更するとリアルタイムに更新されます。

### Theme（Light / Dark / System）

画面全体の配色を切り替えられます。

1. **☰ Menu** → **Theme** 行
2. **☀ Light** / **🌙 Dark** / **🖥 System** のいずれかをクリック

| 項目 | 内容 |
|------|------|
| System | ブラウザ/OS の `prefers-color-scheme` 設定に追従（既定）|
| 保存 | `__settings__` に保存され、リロード後も維持されます |

### Scanline（スキャンライン装飾）

CRT モニター風の走査線を重ねる視覚エフェクト。

1. **☰ Menu** → **Scanline** 行
2. **○ Off** / **◉ On** のどちらかをクリック

### Accent color（アクセントカラー）

選択状態・ボタン・フォーカスリング等のアクセントカラーを変更できます。**Neon Green に戻す** ボタンで既定色に戻ります。

### Border color（ボーダーカラー）

枠線やセパレータの色。**Default** ボタンでテーマ既定色に戻ります。CSS 変数 `--c-border` をオーバーライドします。

### Background color（背景色）

画面全体の背景色を変更できます。**左ペイン（サイドバー）・中央ペイン・右ペイン（メタ情報）のすべてに反映**されます。**Default** ボタンでテーマ既定色に戻ります。CSS 変数 `--c-bg` をオーバーライドします。

### UI Text color（UI テキストカラー）

サイドバー・メニュー・ラベル等の UI 全体の文字色を変更できます。**Default** ボタンでテーマ既定色に戻ります。CSS 変数 `--c-fg` / `--c-text` をオーバーライドします。

### Body Text color（本文テキストカラー）

エントリ本文（Markdown レンダリング、エディタ等）の文字色を変更できます。UI 全体の文字色とは独立です。**Default** ボタンでテーマ既定色に戻ります。CSS 変数 `--c-body-text` をオーバーライドします。

### Font（フォント）

画面全体のフォントファミリーを変更できます。

- **プルダウン**: プリセットから選択（BIZ UDGothic / IBM Plex Mono / Share Tech Mono / Inter / Sans Serif / Monospace）
- **直接入力**: テキスト欄にフォント名を直接入力可能。**直接入力が優先**されます（プルダウンとの併用時）
- **Preset** を選ぶとテーマ既定のフォントスタックに戻ります

| 項目 | 内容 |
|------|------|
| CSS 変数 | `--font-sans` をオーバーライドします |
| 優先順位 | 直接入力 > プルダウン > テーマ既定 |

### Language（言語）

日付フォーマットや `<html>` の `lang` 属性に反映されます。

| 項目 | 内容 |
|------|------|
| 選択肢 | System / 日本語 (ja) / English (en) / English (en-US) / 繁體中文 (zh-Hant-TW) / 한국어 (ko) |
| 反映範囲 | 日付表示のロケール・`<html lang>` 属性 |
| 制限 | **UI 文字列の翻訳は未実装です。**ボタンやラベル等は英語/日本語混在のままです |

### Timezone（タイムゾーン）

日時表示のタイムゾーンを設定します。TEXTLOG のタイムスタンプや Todo の日付表示に反映されます。

| 項目 | 内容 |
|------|------|
| 選択肢 | System / Asia/Tokyo (JST) / UTC / America/New_York (ET) / America/Los_Angeles (PT) / Europe/London (GMT/BST) |
| 反映範囲 | `Intl.DateTimeFormat` の `timeZone` オプション |

> **保存の仕組み**: 外観設定は `__settings__` という**隠しエントリ**（サイドバーには表示されない）に記録されます。エクスポートしたコンテナを別環境で開いても設定が引き継がれます。詳しくは [09 トラブルシューティングと用語集 — System Settings Entry](09_トラブルシューティングと用語集.md#system-settings-entry-__settings__) を参照。
>
> **リードオンリー閲覧中の注意**: Rehydrate 直後などの **read-only** なコンテナ表示中でもすべての設定を変更できますが、その変更はリロードで消えます（コンテナに書き込めないため）。通常の編集可能な状態なら自動保存されます。

## Kanban 列のドロップハイライト

Kanban ビューで Todo カードを列（Open / Done）にドラッグしたとき、ドロップ先の列が薄いアクセントカラー（約 12% 不透明度）のハイライトと破線枠で強調されます。列のタイトルや既存カードのテキストを確認したままドロップできます。

> サイドバーのフォルダへのドラッグ時のハイライトは従来どおり（変更なし）です。

### Kanban カードの選択状態（可読性改善）

以前は Kanban カードを選択したとき、カード全体がアクセントカラーで塗りつぶされてカード内のテキストが読みにくくなる場合がありました。現在は:

- カード全面を**薄いアクセントのティント**（約 14% 不透明度）に変更
- 本文の文字色は通常の色のまま保持
- 選択状態は**アクセント色の枠線 + インセットシャドウ**で表現

サイドバー・Calendar・Slash メニュー等、他のビューの選択表示は従来どおりです（アクセントべた塗り）。

## ドラッグ&ドロップ（Drag & Drop）

エントリの並び替えや関係付けをマウス操作で行えます。用途ごとに 3 つの系統があります。

### サイドバーでのツリー移動

- サイドバーでエントリを別フォルダの上にドロップすると、そのフォルダの子になります（`structural` relation が張り替えられます）
- ルート領域にドロップすればフォルダ配下から外せます
- folder 同士のドロップでフォルダの親子関係も変えられます

### Kanban でのステータス変更

- Todo カードを別の列（Open / Done）にドロップすると、そのままステータスが切り替わります
- 複数選択した状態でドラッグすると、選択中の Todo すべてが一度に移動します

### Calendar での期日変更

- Todo を別の日付セルにドロップすると、その日付が期日として設定されます
- 複数選択状態でのドラッグも同様に全件まとめて期日が変わります

### 注意点

- ドラッグ中はドラッグゴーストに件数バッジが表示され、何件動くかが見えます
- DnD でのフォルダ変更は `Relation` の張り替えであり、エントリ本体の内容は変わりません
- ドロップ先が意図と違っていたら、そのまま Escape でキャンセルできます

## コンテキストメニュー

サイドバーのエントリを右クリックするとコンテキストメニューが開き、よく使う操作にすぐアクセスできます。

- エントリの編集 / 複製 / 削除
- フォルダへの移動
- 別ウィンドウで開く（後述）
- このエントリを単独で持ち出す（selected-only export）

コンテキストメニューから開いた操作は、ツールバーのボタンから行うものと機能的に同じです。

## 別ウィンドウ表示（Entry Window）

エントリを別のブラウザウィンドウで開いて、メインのウィンドウと並べて作業できます。

- **開き方**: エントリをダブルクリック（コンテキストメニューや `Enter` キーからも開けます）
- **用途**: 参照用に常時表示しておきたいメモ、編集とプレビューを並べて見たいドキュメントなど
- **追従**: メイン側で対象エントリや、別エントリに埋め込まれている TODO などが更新されると、別ウィンドウの表示も自動で更新されます
- **閉じ方**: ウィンドウをそのまま閉じます。メインワークスペースの状態には影響しません

### 注意点

- 別ウィンドウは **同じタブ内のポップアップ** としてブラウザに扱われます。ブラウザのポップアップブロックが有効だと開けないことがあります
- ブラウザを丸ごと閉じると、別ウィンドウも閉じます
- 別ウィンドウで編集を確定すると、メインのワークスペースに保存されます。メイン側で同じエントリを並行編集していた場合は、**後の保存が無条件に上書きすることはありません**（後述の「並行編集の安全装置」を参照）

### 別ウィンドウでのキーボードショートカット

別ウィンドウ内では、現時点で次の最小セットのショートカットが使えます。他の全般的なショートカット（ペイン切替、サイドバー矢印移動、`Ctrl+;` の日時挿入など）は今のところメインウィンドウ側でのみ動作し、別ウィンドウへの対応は段階的に追加していく予定です。

| ショートカット | 動作 |
|---------------|------|
| `Ctrl+S` / `Cmd+S` | 編集中の内容を保存（`保存` ボタンと同じ動作）。ブラウザの「名前を付けて保存」は抑制されます |
| `Escape` | 編集中なら編集をキャンセルして閲覧状態に戻す、閲覧状態でそのまま押すと別ウィンドウを閉じる |

## TEXT と TEXTLOG を変換する

TEXT（Markdown ドキュメント）と TEXTLOG（時系列ログ）は、中身を保ったまま相互に変換できます。

### TEXTLOG → TEXT

TEXTLOG のログ行を選んで、1 つの Markdown ドキュメントにまとめます。

1. TEXTLOG を開く
2. **ログを選択** ボタンで選択モードに入る
3. まとめたいログにチェックを入れる
4. **→ TEXT に変換** で preview を表示し、内容を確認
5. **作成** で新しい TEXT エントリが作られる（元の TEXTLOG は変更されません）

### TEXT → TEXTLOG

TEXT の本文を章ごとに区切って、新しい TEXTLOG を作ります。

1. TEXT を開く
2. **→ TEXTLOG に変換** ボタンで preview を表示
3. 区切り方を選ぶ
    - **ATX heading (#, ##, ###)** — 見出しごとに 1 ログ
    - **Horizontal rule (---)** — `---` ごとに 1 ログ
4. タイトルを必要に応じて編集
5. **作成** で新しい TEXTLOG が作られる

### 注意点

- preview を開いている間に別のエントリを選んだり編集モードに入ったりすると、preview は自動で閉じます
- 変換結果には「どの TEXT / TEXTLOG から生まれたか」を示すメタログ / メタリンクが自動で付きます
- 元データは書き換えられません。同じソースから何度でも変換できます

## 一括読み込み（Batch Import）

複数のテキストファイルや Markdown ファイルを、まとめて 1 回の操作で PKC2 に取り込めます。

1. **Batch Import** ボタンからファイル群を選択
2. 読み込み結果が **プレビュー** として表示される
3. 各ファイルを個別にチェックして、本当に取り込むものを選ぶ
4. **取り込み先フォルダ** を任意に指定（未指定ならルート）
5. **Confirm Import** で取り込み実行

- 取り込みは **追加** 操作です。既存のエントリは消えません（通常の Import とは違います）
- 取り込み後は何件成功 / 失敗したかの要約が通知されます

## リンクインデックス

エントリ本文に書いた `entry:<lid>` 形式の参照が、meta pane（右側の情報パネル）で 3 つのセクションに自動整理されます。セクションはエントリを選択したときだけ表示されます。

### 3 つのセクション

| セクション | 内容 |
|---|---|
| **Outgoing links** | 選択中のエントリが参照している他のエントリ一覧 |
| **Backlinks** | 選択中のエントリを参照している他のエントリ一覧 |
| **Broken links** | 参照先エントリが存在しないもの（broken 参照）|

件数はヘッダーの `(N)` に表示されます。参照が 0 件の場合は「No outgoing links.」などの空状態が表示されます。

### クリックで遷移する

**Outgoing links** と **Backlinks** の各行をクリックすると、対象のエントリへ移動します。

**Broken links** の行はクリックしても遷移しません（参照先のエントリが存在しないため）。broken 行は視覚的なマーカーで区別されています。

### どのエントリが対象か

本文をスキャンして outgoing を抽出できる archetype は **text / textlog / folder / todo** の 4 種類です。これら以外（form / attachment / generic / opaque）は outgoing を持ちませんが、他のエントリから参照されていれば backlinks には出現します。

抽出対象は body 内の `entry:<lid>` 参照のみです。Relations（structural / categorical / semantic など）や外部 URL は対象外です。

### self-link と重複参照

- 自分自身への参照（`entry:<自分の lid>`）は Outgoing と Backlinks の両方に出現します
- 同じエントリへの複数参照は 1 件に集約されます

### v1 の非対象

以下は現時点では提供していません:

- グラフ描画
- broken link の自動修復
- 外部 URL の存在確認
- Container 外のエントリへの参照
- Outgoing / Backlinks の並び替え UI

## 関連付け（Relations）

エントリ同士を 4 種類の関係で結びつけられます。

| 種別 | 意味 | 例 |
|------|------|-----|
| **structural** | 親子関係（フォルダ所属） | folder → 子エントリ |
| **categorical** | カテゴリ・タグ | 「技術メモ」タグ |
| **semantic** | 意味的参照 | 「関連するエントリ」 |
| **temporal** | 時間順 | 「前のエントリ → 次のエントリ」 |

タグは `categorical` relation の実装です。関連付けの作成・削除は各エントリの関連情報パネルから行います。

## リビジョン履歴

編集を確定するたびに、編集前の状態がリビジョンとして自動保存されます。選択中のエントリにリビジョンが 1 件以上あると、メタパネルに **Revision history** パネル（折りたたみ式）が表示されます。

### パネルの見方

各行には、そのリビジョンの **記録日時**・**archetype**・**content_hash の先頭 8 文字**（付いていなければ `—`）と、**Restore** / **Restore as branch** の 2 つのボタンが並びます。最新リビジョンが一番上で、下に行くほど古くなります。

### 2 つの復元操作

- **Restore（上書き復元）** — 現在のエントリの本文・タイトルを、その行のスナップショット内容で **上書き**します。上書きする直前の状態は自動で新しいリビジョンとして記録されるので、復元のやり直しも履歴をたどってやり直せます。archetype が異なるリビジョンへの Restore は拒否されます
- **Restore as branch（分岐復元）** — スナップショットの内容を **新しいエントリ**として追加します。元のエントリはそのまま残り、新しい分岐エントリが選択状態になります。分岐元と分岐先は `provenance` リレーションで自動的に紐付けられます（用語集の **Provenance Relation** を参照）

メタパネル上段の **Revert** ボタンは、**最新リビジョン専用**のショートカットとして残っています。中身は「最新リビジョンへの Restore」と同じです。任意の過去リビジョンを対象にしたいときは履歴パネルの `Restore` / `Restore as branch` を使ってください。

### 使える条件・使えない条件

履歴パネルそのものは閲覧できますが、`Restore` / `Restore as branch` のボタンは以下の状況では表示されません（バックエンド側でも同じ条件で拒否されます）:

- **readonly** ワークスペース
- **view-only（埋め込み起動）** のワークスペース
- **編集モード中**（Edit を確定または取り消してから操作してください）
- **Import Preview / Batch Import Preview 表示中**

選択中のエントリにリビジョンが 0 件のときはパネル自体が出ません。

### v1 では提供しないこと

以下は意図的に v1 の対象外です。

- リビジョン間の **差分ビューア**
- リビジョンの **検索・絞り込み**
- **複数エントリ / 複数リビジョン** の一括 Restore
- 分岐と元エントリを統合する **セマンティックマージ**

リビジョンは Container に保存されるため、Export にも含まれます。

## アーカイブ表示

完了した Todo は **アーカイブ** することで日常の視界から隠せます。

- **アーカイブする**: Todo の詳細でアーカイブボタン
- **表示切替**: `showArchived` フラグで一覧に出すかを切り替え
- **Calendar ビュー**: `showArchived` に従う
- **Kanban ビュー**: 常に除外

## 自動保存について

PKC2 は変更のたびに **約 300 ミリ秒後** に IndexedDB へ自動保存します。

- 連続した編集はまとめて 1 回の保存にデバウンスされます
- ブラウザタブを閉じるときに未保存の変更があれば、ブラウザの `pagehide` イベントで強制的に保存されます
- 手動保存ボタンはありません

## 並行編集の安全装置

同じエントリを **別ウィンドウ** や **複数タブ** などで同時に編集し、両方が保存しようとすると、PKC2 は後から来た保存を **自動で止めて専用ダイアログを表示**します。知らないうちに片方の編集内容が消えることはありません。

### いつダイアログが出るか

編集モードに入った瞬間、PKC2 はそのエントリの「編集開始時のスナップショット」を記録します。その後 `Ctrl+S` / `Cmd+S` や保存ボタンで確定しようとした時点で、エントリの現在の状態がスナップショットと違っていれば、保存は中止されてダイアログが開きます。具体的には次のいずれかです。

- 編集中に別ウィンドウ側が先に保存を確定した（並行編集）
- 編集中にそのエントリ自体が削除された
- 編集中にそのエントリの archetype が切り替わった

### ダイアログで選べる 3 つの操作

ダイアログには以下の 3 つのボタンがあり、**Save as branch** に初期フォーカスが当たります（最も安全な選択肢だからです）。

| ボタン | 動作 |
|--------|------|
| **Save as branch** | 編集内容を **新しいエントリ** として追加し、元のエントリは最新の状態のまま残します。元と新のエントリは `provenance` リレーションで「どちらから派生したか」が記録されます |
| **Discard my edits** | 編集内容を **破棄** します。元のエントリは最新の状態のまま残り、自分が書いていたドラフトは失われます |
| **Copy to clipboard** | 編集内容の本文をクリップボードにコピーします。**ダイアログは閉じず**、上の 2 つの操作で後始末してから別の場所に貼り直せます |

### ダイアログの挙動

- **ダイアログは勝手には閉じません**。`Escape` を押しても、ダイアログの外をクリックしても閉じません。編集内容を守るために、必ず **Save as branch** / **Discard my edits** のどちらかを明示的に選んでください
- **元のエントリは保護されます**。ダイアログが出た時点で自分の保存は中止されており、元のエントリは相手側の最新状態のままです。ここで **Save as branch** を選んでも元エントリを上書きしません
- **編集中のドラフトは保持されます**。ダイアログが出ている間、入力したタイトル・本文・添付は内部に保持されているので、**Copy to clipboard** で外に逃がすこともできます

### v1 で提供しないこと

以下は意図的に v1 の対象外です。

- **リアルタイム同期** — 2 つのウィンドウが同じエントリを開いていても、編集中の内容は互いに共有されません。保存時点で初めて検出します
- **自動マージ** — 両方の編集内容を自動でマージする機能はありません。どちらを残すかは必ずユーザーが選びます
- **差分ビューア** — ダイアログ内で相手の変更内容を行単位で比較する UI は持ちません。相手側の本文は通常どおり別ウィンドウやリビジョン履歴で確認してください
- **編集開始時の事前警告バナー** — 「このエントリは別セッションから編集中です」といった予告は v1 では出ません。検出は保存時に行われます
