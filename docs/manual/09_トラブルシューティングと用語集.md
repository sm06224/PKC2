# トラブルシューティングと用語集

## トラブルシューティング

### データが消えた・見当たらない

**症状**: 前回作成したエントリが表示されない。

**考えられる原因**:

1. **別のブラウザ / 別の HTML ファイルで開いている**: IndexedDB はブラウザごと・URL ごとに分かれています。同じ HTML ファイルを同じブラウザで開き直してください
2. **ブラウザのサイトデータをクリアした**: ブラウザの履歴・サイトデータ削除で IndexedDB も消えます
3. **プライベートブラウジングで使った**: プライベートモードのデータはタブを閉じると消えます
4. **ファイルのダウンロード先が変わった**: 同じ HTML ファイルでも保存場所が変わると `file:///` URL が変わり、別のストアになります

**対処**:

- データの保険として定期的に Export（特に ZIP Package）でバックアップしてください
- 複数デバイスで使う場合は、ZIP Package を共有ストレージに置いて運用するのが確実です

### Readonly モードで編集できない

**症状**: エントリの編集ボタンが表示されない。

**原因**: Readonly としてエクスポートされた HTML を開いています。

**対処**:

- 編集したい場合は **Rehydrate to Workspace** ボタンを押してください
- Rehydrate すると新しい Container ID が発行され、編集可能なワークスペースとして IndexedDB に保存されます
- 元の HTML ファイル自体は変更されません

### ファイルサイズが大きいと警告が出る

**症状**: ファイル添付時に「このファイルは大きすぎます」的なメッセージ。

**原因と対処**:

- **1 MB 以上**: 軽い警告。操作はブロックされません
- **5 MB 以上**: 重めの警告。Export の HTML サイズに影響します。大きい添付が多い場合は **ZIP Package** でのバックアップを検討してください
- これらは制限ではなく情報提供です。無視して進めることもできます

### Export したら HTML が数十 MB になった

**症状**: 書き出した HTML が重い。

**原因**: 添付ファイルを HTML Full モードで同梱すると、gzip 圧縮しても一定のサイズになります。

**対処**:

- 共有用途なら **HTML Light**（添付実体なし）を使う
- バックアップ用途なら **ZIP Package**（base64 膨張なし）を使う
- 巨大な添付は元ファイルを別管理にする選択も検討

### Import したらデータが全部置き換わった

**症状**: Import 後に元のエントリが消えた。

**原因**: Import preview で **Replace** を選ぶと HTML / ZIP の Import は全置換です。データを保持したまま追加取り込みしたい場合は **Merge** を選ぶ必要があります。

**対処**:

- Import 前に必ず Export でバックアップを取る習慣をつけてください
- プレビュー上部のラジオで **Merge** を選べば既存データを保持したまま追加できます（同名 entry は conflict 解決 UI で個別に処理）
- Confirm Import / Confirm merge のダイアログで内容を確認してから確定しましょう
- テキスト / Markdown を単純に **追加** したいだけなら、**Batch Import** や **単体バンドル（.text.zip / .textlog.zip）** を使う方が目的に合います
- Merge mode の詳細は [07 保存と持ち出し — Merge mode と conflict 解決 UI](07_保存と持ち出し.md) を参照してください

### Merge import の Confirm merge が押せない

**症状**: Import preview で **Merge** を選んでいるが、下段の **Confirm merge** ボタンがグレーアウトで押せない。

**原因**: host 側と imported 側で同名 entry が検出され、**conflict 解決 UI** に未選択の行が残っています。Confirm merge はすべての C2 / C2-multi 行がラジオ選択済みになるまで disabled のままです。

**対処**:

- プレビュー中段の **Entry conflicts: N** セクションをスクロールし、⚠ バッジの行（C2 / C2-multi）で Keep current / Duplicate as branch / Skip のいずれかを選んでください
- **C2-multi** 行（`⚠ N host candidates` バッジ）では Keep current が **disabled** になっています。host 側候補が複数あり「どれを残すか」が曖昧なためで、Duplicate as branch か Skip を選んでください
- 全件まとめて片付けたい場合は conflict セクション下部の **Accept all host**（C1 / C2 を Keep current に一括）または **Duplicate all**（全件を Duplicate as branch に一括）を使うと早いです
- Confirm merge 近傍に `Resolve N pending conflicts` と残件数が表示されます。N が 0 になれば押せます

### Merge に切り替えたのに conflict UI が出ない

**症状**: Import preview で **Merge** を選んでも Entry conflicts セクションが現れない。

**考えられる原因**:

1. **衝突が 0 件** — host と imported で同じ archetype + 正規化タイトルの entry が無ければ UI は表示されません。そのまま Confirm merge で追加取り込みできます
2. **Replace を選んだまま** — conflict UI は Merge mode 専用です。プレビュー上部のラジオで **Merge** に切り替わっているか確認してください
3. **schema version が一致していない** — host と imported の `schema_version` が違うと、conflict 検出はスキップされ Import 自体も schema gate で reject されることがあります。プレビュー画面のヘッダー表示を確認してください
4. **タイトル正規化で別物と判定された** — タイトル比較は NFC + 空白圧縮のみで、大文字小文字やピリオド／記号の差は別タイトル扱いです。期待した衝突が検出されない場合はタイトル文字列を確認してください

### Merge で選んだ conflict の判断が Cancel 後に消えた

**症状**: conflict UI で一通りラジオを選んだ後に Cancel merge を押したら、再度 Merge に切り替えたときに選択が戻っていない。

**原因**: **意図的な仕様** です。conflict の意思決定は 1 プレビューセッション限りで、以下のタイミングで破棄されます。

- Cancel merge を押した
- Replace に切り替えた
- 別ファイルで再度 Import preview を開いた

**対処**: 判断をやり直したくない場合は Cancel を押さずに Confirm merge まで進めてください。途中で Replace に切り替えるつもりが無いなら Merge のまま置いておいて問題ありません。

### ZIP Import 後に「警告あり」の toast が出た

**症状**: `.pkc2.zip` の Import 直後に「ZIP import succeeded with a warning: ...」や「N warnings」といった黄色い toast が出る。

**原因**: ZIP の中身に不整合があり、PKC2 側で安全に解決したことを知らせる通知です。**Import 自体は成功しています**。次のような状況で出ます。

- 同じ asset key が複数回入っていた
- `manifest.json` や `container.json` が複数入っていた
- `/` や `..` を含む不正な asset key があった

**対処**:

- 心当たりのある ZIP（例: 2 つの ZIP を手動でマージしたもの）であれば、多くの場合そのまま進めて大丈夫です
- 詳細を知りたい場合はブラウザのデベロッパーツール（`F12`）のコンソールを開いて `[PKC2] ZIP import warning: ...` の行を確認してください
- 出どころ不明の ZIP の場合は、toast を閉じずに一度内容を確認してから進めるとよいです

### IndexedDB が壊れた・読み込みエラー

**症状**: PKC2 を開いても初期化が終わらない、エラーが出る。

**対処**:

1. ブラウザのデベロッパーツール（`F12`）を開き、コンソールのエラーメッセージを確認
2. **Clear IDB** ボタンが表示されている場合はそれを使って IndexedDB をクリア（データは消えます）
3. IDB クリア後、HTML 内の pkc-data にフォールバックします。Export 済みのバックアップから Import で復元してください
4. 再発する場合は別のブラウザで試してみてください

### ブラウザのタブを閉じたら保存されなかった気がする

**症状**: 編集中にタブを閉じたら最新の変更が消えた。

**対処**:

- PKC2 は 300 ミリ秒のデバウンスで保存するため、編集直後にタブを閉じると間に合わないことがあります
- 確実に保存したい場合は `Ctrl+S` / `Cmd+S` で明示的に確定してください
- `pagehide` イベントでの強制保存もありますが、OS の強制終了では働きません

### 別ウィンドウ（Entry Window）が開かない

**症状**: エントリをダブルクリックしても別ウィンドウが開かない。

**対処**:

- ブラウザのポップアップブロックが有効だと開けません。アドレスバー付近のポップアップ許可アイコンから PKC2 を許可してください
- 別ウィンドウは `window.open` を使います。ブラウザ設定で無効化されているケースでは動きません

### 別ウィンドウが「古い内容」のままになる

**症状**: メインワークスペースで Todo を更新したのに、別ウィンドウ内で表示している埋め込みカードや本文が古いまま。

**対処**:

- 通常は自動で追従します。追従しない稀なケースでは、別ウィンドウをいったん閉じて再度開き直してください
- これは PKC2 の既知の追従経路で、メイン側の更新が別ウィンドウに伝わるようになっています

### 埋め込み（iframe）で PKC2 が動かない

**症状**: 親ページに iframe で PKC2 を埋め込んだが、表示されない・操作できない。

**対処**:

- iframe の `sandbox` 属性に `allow-scripts allow-same-origin` が含まれているか確認
- `allow-top-navigation` は付けないでください（セキュリティ境界を壊します）
- 親ページから `postMessage` で通信する場合は `origin` を検証してください
- 詳細は開発者向け文書 `docs/planning/14_基盤方針追補_clone_embed_message.md` を参照

### Replace（置換）ボタンが見当たらない

**症状**: 編集中の action bar に `🔎 Replace` ボタンが出ない。

**考えられる原因**:

1. **編集モードに入っていない** — まず `✏️ Edit` を押して編集モードに遷移してください
2. **対象エントリが TEXT ではない** — Replace は TEXT エントリの本文に限定です。textlog / form / attachment / folder は対象外
3. **Readonly モードで開いている** — Rehydrate してから編集してください（「Readonly モードで編集できない」を参照）
4. **Historical revision を閲覧中** — 履歴スナップショットは読み取り専用です。現在の状態に反映したい場合は **復元** を使ってください

### 置換ダイアログの Apply が押せない

**症状**: ダイアログを開いて検索文字列を入れたのに、`Apply` ボタンがグレーアウトしたままで押せない。

**考えられる原因**:

- **Find 欄が空** — 検索文字列を入力してください
- **Regex が有効で、正規表現として無効** — 状態行に「Invalid regex: ...」が赤文字で表示されます。パターンを直すか、Regex チェックを外してください
- **ヒット 0 件** — 状態行に「No matches in current entry.」と出ている場合は、本文中に一致する文字列がありません

### 置換ダイアログの Regex で思った通りに置換できない

**症状**: `$1` や `\d+` を期待通り動かしたい。

**対処**:

- **Regex チェックを ON** にしていないと、Find 欄は **リテラル文字列** として扱われます。`\d+` は `\d+` という 3 文字そのものの検索になります
- 置換文字列（Replace with）で `$1` / `$&` / `$<name>` などの back-reference を使うには、**Regex チェックが ON** である必要があります。OFF の時は `$1` はリテラル文字列としてそのまま挿入されます

### リンクをコピーして貼り付けても URL が残らない

**症状**: ブラウザから「`Example`」というリンクをコピーしたはずなのに、本文にはラベルだけが入り URL が消える。

**考えられる原因**:

1. **貼り付け先が対象外の textarea** — タイトル欄・form などは対象外です。対象は TEXT エントリの本文 / TEXTLOG の追記欄（textlog-append-text）・ログ入力欄（textlog-entry-text）です
2. **編集モードに入っていない** — TEXT または TEXTLOG エントリを編集モードで開いてから貼り付けてください
3. **クリップボードに text/html が含まれていない** — コピー元がプレーンテキストしか提供していないケースです。この場合はラベルだけが貼り付けられます
4. **URL が危険なスキーム** — `javascript:` / `data:` / `vbscript:` の URL は安全のためリンク化されず、ラベルのみプレーンテキストとして残ります

**対処**: 対象 textarea の編集モードで貼り付け、かつコピー元が text/html 形式のリンクを提供していれば、自動で `[ラベル](URL)` の形に整形されます。

### 貼り付けたら `[URL](URL)` ではなく URL だけになった

**症状**: リンクをコピーして貼り付けたら、`[https://example.com](https://example.com)` のような形ではなく、`https://example.com` だけが挿入された。

**原因**: これは **意図した動作**（bare URL 挿入）です。クリップボード上でリンクのラベルと URL が同じ文字列になっているとき（アドレスバーから URL をそのままコピーした場合など）、PKC2 は冗長な `[URL](URL)` を避けて **bare URL** だけを挿入します。Markdown では bare URL はそのままリンクとして機能します。

ラベルが URL と異なる場合（例: 「Example Page」というラベルで `https://example.com` をコピー）は、従来通り `[Example Page](https://example.com)` の形に整形されます。

### TEXTLOG でリンクを貼り付けても `[ラベル](URL)` にならない

**症状**: TEXTLOG の追記欄やログ入力欄にリンクを貼り付けたが、Markdown リンクに変換されない。

**考えられる原因**:

1. **TEXTLOG のタイトル欄に貼り付けている** — タイトル欄は対象外です。追記欄（textlog-append-text）またはログ入力欄（textlog-entry-text）に貼り付けてください
2. **編集モードに入っていない** — TEXTLOG エントリを `✏️ Edit` で編集モードにしてから貼り付けてください
3. **クリップボードに text/html が含まれていない** — プレーンテキストだけのコピーでは変換されません（上記「リンクをコピーして貼り付けても URL が残らない」と同様）

**対処**: TEXTLOG の追記欄・ログ入力欄は TEXT 本文と同様にリンク変換の対象です。編集モードで正しい textarea に貼り付けることを確認してください。

### Firefox などでアドレスバーコピーが Markdown リンクにならない（表示時はリンクになる）

**症状**: Firefox の macOS 版などで、アドレスバーから URL をコピーして TEXT / TEXTLOG に貼り付けたが、`[ラベル](URL)` の形に自動整形されず、URL 文字列がそのまま入る。

**原因**: ブラウザによってはアドレスバーコピー時にクリップボードへ `text/plain` しか書き込まないものがあります（Firefox macOS など）。この場合、貼り付け時の自動整形（上記「リンクをコピーして貼り付けても URL が残らない」で説明している経路）は、コピー元側にリンクの HTML 情報がないため起動しません。結果として textarea には bare URL がそのまま入ります。

**挙動**: PKC2 はこの状況を **表示時の autolink fallback** で救済しています。

- **保存される本文**: 貼り付けた bare URL のまま（`https://example.com`）
- **read 表示（閲覧モード）**: URL 部分が `<a>` でクリック可能に描画されます（`target="_blank"` / `rel="noopener noreferrer"`）
- **編集モード（textarea）**: 文字列としての URL がそのまま見えます（`<a>` タグはテキスト欄に混ざりません）

つまり **「本文はただの URL のままだが、読むときはリンクとして見える」** 状態になります。新しく外部からタイトル情報を取得したり、保存データを書き換えたりはしません。

**対処**:

- 保存データを `[ラベル](URL)` の形で持ちたい場合は、手動で `[ラベル](URL)` の Markdown 記法で入力してください
- 表示上クリックできれば十分な場合は、そのまま bare URL を貼り付けて保存して問題ありません
- タイトルを外部から自動で引っ張ってくる（OGP 取得のような rich preview）は本バージョンでは実装していません

**安全**: `javascript:` / `file://` / `data:text/html` のような危険スキームは、bare text に書かれていても autolink されません。そのままテキストとして表示されます。

### Export した HTML を開いたら「起動元を選ぶダイアログ」が出た

**症状**: Export HTML を開いた瞬間に「どちらのコンテナを開きますか？」のダイアログが表示される。

**原因**: HTML 内部の埋め込みコンテナ（pkc-data）と、そのブラウザに保存済みの IndexedDB のワークスペースの **両方が見つかった** ためです。どちらを表示するかはユーザーに選ばせる仕様です。

**対処**:

- **Export の中身を見たいだけ**: 「埋め込みを開く (view-only)」を選択。編集しても IDB には保存されません（タブを閉じれば元のワークスペースに戻ります）
- **普段使いのワークスペースを続けたい**: 「IndexedDB を開く」を選択。埋め込みコンテナは無視され、手元の IDB データがそのまま使われます
- このダイアログは Escape や背景クリックでは閉じません。必ずどちらかを選んでください

詳しくは [07 保存と持ち出し — HTML を開いたときの起動元](07_保存と持ち出し.md) を参照してください。

### HTML を開いて編集したのに、保存されず消えた

**症状**: Export HTML を開いて編集したが、タブを閉じてから開き直すと編集内容が消えている。

**原因**: 埋め込みコンテナから起動した（view-only）状態では、**IndexedDB への自動保存が行われません**。これは「受信者が Export を見るだけで自分のワークスペースが上書きされる」事故を防ぐための仕様です。編集は画面上のみで、リロードで消えます。

**対処**:

- そのセッションだけでよい確認用途なら、これは意図した挙動です
- 編集を残したい場合は、**明示的に Import する** 必要があります:
  1. 現在の HTML をブラウザの標準機能（`Ctrl+S` / 右クリック→保存）でファイル保存
  2. **Data…** パネルの **Import** ボタンからそのファイルを選ぶ
  3. プレビュー→ **Confirm Import** で IndexedDB に取り込まれます
- Import 確定後は通常のワークスペースとして自動保存されます

### Export した HTML を開くと、手元の IndexedDB ではなく Export 側の内容が表示される

**症状**: Export HTML を開いた時、いつも使っているワークスペースではなく、Export 時点のスナップショットが表示される。ダイアログは出ない。

**原因**: そのブラウザに PKC2 の IndexedDB がまだ無い（初回 / 別ブラウザ / サイトデータクリア後 など）ため、埋め込みコンテナから起動しています。両方が揃っていればダイアログが出ますが、片側しかない場合は自動的にある方から起動します。

**対処**:

- 通常はこれが期待動作です。Export の中身をそのまま見たい用途に向いています
- この状態は **view-only** なので、編集しても IndexedDB には保存されません。編集を残したい場合は上記「HTML を開いて編集したのに、保存されず消えた」の Import 手順を使ってください
- 普段のワークスペースを別ブラウザで作り直したい場合は、空の HTML（生の `pkc2.html`）を開くか、Export ファイルを Import してください

### TEXTLOG のログ行に `🔎` ボタンが見当たらない

**症状**: textlog を編集モードにしたのに、各ログ行に `🔎` ボタンが現れない。

**考えられる原因**:

1. **編集モードに入っていない** — まず textlog の `✏️ Edit` を押してください。viewer（閲覧モード）では表示されません
2. **Readonly ワークスペースを開いている** — 「Readonly モードで編集できない」を参照
3. **Historical revision を閲覧中** — 履歴スナップショットは読み取り専用で、編集モードに入れません
4. **対象が textlog ではない** — `🔎` ボタンは textlog entry の各ログ行にのみ表示されます。TEXT の本文には action bar 側の `🔎 Replace` ボタンが別途あります

### TEXTLOG のログ行置換ダイアログの Apply が押せない

**症状**: ログ行の置換ダイアログを開いたのに、`Apply` がグレーアウトしている。

**考えられる原因と対処**:

- **Find 欄が空** — 検索文字列を入力してください
- **Regex が有効で、無効な正規表現** — 状態行に `Invalid regex: ...` が赤文字で表示されます。パターンを直すか Regex チェックを外してください
- **ヒット 0 件** — 状態行に `No matches in current log.` と出ている場合、そのログ行の本文中に一致する文字列がありません。他のログ行を対象にしたい場合はいったん Close して別の行の `🔎` を押してください

### TEXTLOG の複数ログをまとめて置換したい

**結論**: v1 では提供していません（意図的な仕様）。

**理由**: ログ単位の独立性を守るためです。時刻・フラグ・並び順・ID を壊すリスクが高く、慎重に設計する必要があるため段階的に解放します。

**対処**:

- 今は各ログ行の `🔎` を 1 件ずつ押して順番に置換してください
- 将来の拡張（複数ログ / 全 textlog 一括置換）は別テーマとして検討されています

### TEXTLOG でログ行を置換したが、時刻やフラグは変わらない

**症状**: ログ行の本文を Apply で置換したが、タイムスタンプや ★フラグ、ログ ID は変わらなかった。

**結論**: これは **意図的な仕様** です。

**理由**: ログ行の `text` のみが置換の対象で、時刻 / フラグ / ID / 並び順 / 他のログ行は構造的に保護されています。時刻やフラグを変更したい場合は、編集モードの該当行の timestamp span（読み取り専用）ではなく、チェックボックスや別 UI を使うか、削除 + 再追記で対応してください。

### TEXTLOG のログセルに画像をペーストしたら、別のセルに入ってしまった

**症状**: TEXTLOG 編集モードで 2 件目以降のログセルにフォーカスして画像をペーストしたのに、先頭ログに挿入された。

**期待動作**: フォーカスしているログセルにのみアセットリンクが挿入されます。先頭ログや他のセルに書き込まれることはありません。複数ログが存在する状態でも、ペースト先は常にフォーカス中のセルです。

**対処**: 現在のバージョンではこの症状は発生しません。再発する場合はブラウザのキャッシュをクリアするか、最新の HTML に差し替えてください。詳しくは [05 日常操作 — TEXTLOG への画像貼り付け](05_日常操作.md) を参照してください。

### 画像を貼り付けたらファイルサイズが小さくなった / 形式が WebP に変わった

**症状**: スクリーンショットや JPEG / PNG を貼り付けたら、添付のサイズが元より小さく、MIME が `image/webp` になっている。

**原因**: PKC2 の **画像自動最適化**（Image Intake Optimization）が働いています。512 KB 以上の対応画像（JPEG / PNG / WebP / BMP）は、保存前に WebP 変換 + 長辺 2560 px 縮小で最適化されます。

**対処**:

- 確認ダイアログで **元のまま保存** を選べば、最適化をスキップして原画のまま保存できます
- **原画も保持する** チェックを ON にすれば、最適化後の画像と原画の両方を保存できます（デュアル保存）
- 「次回から確認しない」を選んだ場合は、localStorage を手動で削除すると確認ダイアログが復活します（後述の「remembered choice を解除したい」を参照）

### 原画も残したい

**症状**: 最適化は使いたいが、念のため元の画像も保持しておきたい。

**対処**: 確認ダイアログで **原画も保持する** チェックボックスを ON にして「最適化して保存」を押してください。最適化後の画像が表示用として保存され、原画はアセットストアに `{asset_key}__original` として追加保存されます。エクスポート時に両方が含まれます。

### 透過 PNG や GIF が最適化されない

**症状**: 透過のある PNG をペーストしたが、最適化の確認ダイアログが出ない。GIF も同様。

**原因**: これは **意図した動作** です。

- **透過 PNG** — WebP 変換で透過チャンネルが失われるのを防ぐため、自動でスキップされます。トーストに「透過画像のため最適化をスキップしました」と表示されます
- **GIF** — アニメーションの可能性があり、Canvas 変換で壊れるため、最適化対象外です。そのまま保存されます
- **SVG** — ベクター形式はラスタ変換すると品質が低下するため、対象外です

### remembered choice（記憶設定）を解除したい

**症状**: 画像最適化の確認ダイアログで「次回から確認しない」を選んだが、やり直したい。

**対処**: ブラウザの DevTools（`F12`）→ Application → Local Storage で、以下のキーを削除してください。

- `pkc2.imageOptimize.preference.paste` — 貼り付けの記憶
- `pkc2.imageOptimize.preference.drop` — ドロップの記憶
- `pkc2.imageOptimize.preference.attach` — ボタン添付の記憶

すべて削除すると、次回からすべての操作経路で確認ダイアログが再表示されます。特定の経路だけ解除したい場合は、該当キーだけを削除してください。

### 画像の多い TEXTLOG を開いたら、最初に一部のログしか画像が出ない

**症状**: 画像を多数含む TEXTLOG を開いた直後、上の方のログにだけ画像が表示され、下のログは **タイムスタンプと空白だけ** のカードが並んでいる。

**原因**: これは **意図した動作（Staged Render）** です。PKC2 は画像多数の TEXTLOG で初期表示を軽くするため、先頭 8 件のログ行だけを即時描画し、9 件目以降は placeholder として保留します。

**対処**: スクロールすれば自動的に展開されます（hydrate）。placeholder が画面下端まで約 400 px まで近づいた時点で、そのログが実体化し、画像と本文テキストが描画されます。

- 下にスクロールするだけで順次表示されます
- 一度展開されたログはそのまま保持され、再スクロールで消えることはありません
- 印刷 / PDF エクスポート時は `beforeprint` で全ログを即時展開するため、出力に placeholder が残ることはありません

詳しくは [05 日常操作 — TEXTLOG の画像の段階的表示（Staged Render）](05_日常操作.md) を参照してください。

### TEXTLOG の一部ログに画像が出ておらず、スクロールしてから表示された

**症状**: TEXTLOG で特定のログを見ようと Ctrl+F などで探したら、最初は placeholder（空白カード）のままだった。スクロールして位置を合わせたら画像が描画された。

**原因**: 上記の Staged Render による保留状態です。placeholder には **タイムスタンプ・重要フラグ・行リンクボタン** は描画されているため、検索や行リンクでの特定自体は可能です。viewport 付近に入った時点で自動 hydrate します。

**対処**:

- 目的のログを開きたい場合はそのあたりまでスクロールしてください（近づいただけで hydrate します）
- 印刷プレビューや PDF エクスポートを使うと、TEXTLOG 全体が一括展開された状態で出力されます

### TEXTLOG → TEXT 変換の選択モードで、下にスクロールしたら Placeholder のチェックが外れていた

**症状**: TEXTLOG → TEXT 変換のため複数ログを選択した後に画面をスクロールしたら、途中のログのチェックが外れていないか心配になる。

**期待動作**: Staged Render の placeholder にも選択チェックボックスが描画されており、**選択状態は DOM 置き換えを経ても維持** されます（状態は AppState に保持）。placeholder のままでも checked 表示、hydrate 後の実ログでも checked 表示です。

**対処**: 実際にはチェックは外れていません。念のため確認したい場合は、[05 日常操作 — TEXT と TEXTLOG を変換する](05_日常操作.md) を参照してください。

### 複数ファイルをドロップしたのに一部しか追加されていない

**症状**: 3 ファイルをまとめてドロップしたが、サイドバーに 2 件しか表示されない。

**考えられる原因**:

1. **1 件が読み込みエラーになった** — ファイル読み込みに失敗したファイルはスキップされますが、残りは処理されます。ブラウザのコンソール（`F12`）に `Failed to read "ファイル名"` のログが出ていれば該当します
2. **1 件がサイズ超過警告で止まった** — サイズ超過警告が出た場合でも処理は継続します。別件の警告トーストを見落としていないか確認してください
3. **サイドバーの表示フィルタが絞り込まれている** — 種別フィルタや検索で attachment が非表示になっている可能性があります。フィルタをリセットしてください

**対処**: 読み込み失敗があれば、そのファイルを単独でもう一度ドロップしてください。

### 重複ファイルのトーストが出たが、添付が増えた。意図した動作か

**症状**: `「ファイル名」は既存の添付と同一内容です` というトーストが出たにもかかわらず、サイドバーに attachment エントリが追加された。

**回答**: これは **意図した動作**（情報通知のみ、添付は止めない）です。

PKC2 の重複検知は「同一ファイルをドロップしたことを知らせる」ことが目的であり、添付をキャンセルする機能ではありません。意図的に同一ファイルを複数添付したい場合もそのまま実行できます。attachment エントリはそれぞれ独立したエントリとして保存されます。

### editing 中にサイドバーのドロップゾーンに反応しない

**症状**: エントリを編集モードで開いているときに、サイドバー最下部の「📎 Drop files here」にファイルをドロップしても何も起きない。

**原因**: これは **意図した動作**（editing フェーズ中は非アクティブ）です。

サイドバー常設ドロップゾーンは **ready フェーズ専用** です。editing フェーズ中はゾーンが薄く表示され、ドロップを受け付けません。editing 中にファイルを添付したい場合は、センターペインの編集エリア（body textarea が表示されているエリア）にドロップしてください（詳しくは「編集中のドラッグ&ドロップ / ボタン添付」を参照）。

### センターペインの DnD とサイドバーの DnD はどう違うか

| 項目 | センターペイン DnD | サイドバー常設 DnD |
|------|-----------------|-----------------|
| アクティブな phase | ready または editing（editing 中は自動リンク挿入） | ready のみ |
| 自動リンク挿入 | editing 中の本文 textarea にフォーカスがあれば挿入 | 挿入しない |
| 複数ファイル（ready 時） | 順番に attachment エントリを作成 | 同じ |
| 複数ファイル（editing 時） | 順番に attachment エントリを作成し、各リンクを挿入 | 非アクティブ |

どちらのゾーンも、ready フェーズでのドロップは同じ処理です。editing フェーズのファイル添付にリンク挿入が必要なときはセンターペイン側を使ってください。

### ファイルをドロップ / ボタン添付したのに本文へリンクが入らない

**症状**: TEXT / TEXTLOG / FOLDER の編集中にセンターペインへファイルをドラッグ&ドロップ、または **📎 File** ボタンでファイルを選択したが、attachment エントリは作成されたのに本文 textarea にリンク（`![名前](asset:...)` / `[名前](asset:...)`）が挿入されなかった。

**考えられる原因**:

1. **編集モードに入っていない** — attachment エントリは作成されますが、リンク挿入は editing phase 専用です。まず対象エントリの `✏️ Edit` を押して編集モードに入ってから操作してください
2. **本文 textarea にフォーカスが当たっていない** — ドロップやボタン押下の直前に、どの textarea もクリックしていない場合、挿入先が確定しないためリンク挿入をスキップします。操作前に本文 textarea をクリックしてカーソルを置いてください
3. **TEXTLOG で複数のログセルがあり、どれにもフォーカスがない** — TEXTLOG 編集中にセル外をクリックしてフォーカスが外れた状態でドロップすると挿入しません。目的のセルをクリックしてからドロップしてください
4. **todo / form / attachment archetype を編集中** — これらの archetype は対象外です。attachment エントリのみ作成されます

**確認方法**: ドロップ前に本文 textarea をクリックして、カーソルが点滅していることを確認してからドロップしてください。

### ドロップしたら TEXTLOG の意図しないセルにリンクが入った

**症状**: TEXTLOG 編集中にドロップしたら、フォーカスしていたのとは別のログセルにリンクが挿入された。

**期待動作**: リンクは**操作開始時点でフォーカスしていたログセルにだけ**挿入されます。他のセルは変更されません。

**対処**: 現在のバージョンではこの症状は発生しません。再発する場合はブラウザのキャッシュをクリアするか、最新の HTML に差し替えてください。詳しくは [05 日常操作 — 編集中のドラッグ&ドロップ / ボタン添付](05_日常操作.md) を参照してください。

### TEXT / TEXTLOG の変換プレビューが勝手に閉じた

**症状**: プレビューを開いていたのに、エントリを選び直したり編集モードに入ったら消えた。

**原因**: プレビューは「今選んでいるエントリ」に紐づく一時 UI です。別のエントリを選んだり編集モードに入ると、整合性を保つため自動で閉じられます。

**対処**:

- 変換を完了させたい場合は preview 内で **作成** を先に押してください
- いったん閉じた後に再度開くこともできます（元エントリは変更されません）

### リンクインデックス（Outgoing / Backlinks / Broken）が表示されない

**症状**: meta pane に Outgoing links / Backlinks / Broken links セクションが出ない。

**考えられる原因**:

1. **エントリを選択していない** — セクションはエントリ選択中のみ表示されます。サイドバーでエントリをクリックして選択してください
2. **Container がロードされていない** — Container が null の状態（初期化中やエラー時）ではセクションは出ません
3. **meta pane 自体が折りたたまれている** — 右端の META バーをクリックして展開してください

### Backlinks が出ない

**症状**: 自分のエントリを参照しているはずのエントリがあるのに、Backlinks が空になっている。

**考えられる原因**:

1. **参照元が `entry:` scheme を使っていない** — Relations（フォルダ移動・タグ付け・semantic relation）は Backlinks に含まれません。`entry:<lid>` の形で本文に書いてある参照のみが対象です
2. **参照元の archetype が対象外** — `form` / `attachment` / `generic` / `opaque` の本文はスキャンされません。参照元が text / textlog / folder / todo であることを確認してください
3. **lid が合っていない** — `entry:<lid>` の `<lid>` は大文字小文字を区別します。エントリの LID を確認してください

### Broken links が表示される

**症状**: Outgoing links または Broken links セクションに、行をクリックできない参照が出る。

**原因**: 本文内の `entry:<lid>` で、その `<lid>` に対応するエントリが Container 内に存在しない場合、**broken link** として表示されます。

**よくあるケース**:

- 参照先のエントリを削除した
- リビジョン復元で本文が古い状態に戻り、当時参照していたエントリがもう無い
- 別の Container からコピーした本文に残っている参照
- `<lid>` の打ち間違い

**対処**: 参照先のエントリを再作成するか、本文の `entry:<lid>` を修正してください。broken link は Container 全体の整合性に影響しますが、データの損壊ではなく「参照先が見つからない」状態です。

### 複数のエントリが同じエントリを参照していると Backlinks が多く表示される

**症状**: 特定のエントリを開くと Backlinks に多くの行が並ぶ。

**原因**: これは正常な動作です。選択中のエントリを参照しているエントリが多いほど、Backlinks の件数が増えます（「ハブ」になっているエントリです）。

リストの並びは `container.entries` の配列順（主に作成日時順）に従います。

### サイドバーの ↑ / ↓ ボタンが表示されない

**症状**: Manual で並び替えたいのに、エントリを選んでも ↑ / ↓ ボタンが出てこない。

**原因**: ボタンは表示条件が揃ったときだけ出る設計です。よくある取りこぼしは以下です。

- 並び順セレクタが **Manual** 以外（Title / Created / Updated）になっている
- 現在のビューが **Calendar** または **Kanban**（Detail でのみ出ます）
- エントリが選択されていない（クリックして選択状態にしてください）
- 閲覧専用モードで開いている
- Import プレビューを開いている（閉じるか確定してください）

**対処**: 並び順を Manual に切り替え、Detail ビューで対象エントリを選び直してください。それでも出ない場合は Import プレビューや閲覧専用モードを確認します。

### サイドバーで ↑ / ↓ を押しても動かない

**症状**: ボタンは押せるが順序が変わらない。

**原因**: 対象エントリがすでに **先頭 / 末尾** にいるときは、端を越える方向のボタンは何もしません（エラーにもなりません）。また、フィルタや検索で非表示のエントリが間に挟まっている場合、Container 上の直前 / 直後と入れ替わるため、画面では見た目が変わらないことがあります。

**対処**:

- 反対側のボタンを試してください（先頭なら ↓、末尾なら ↑）
- フィルタや検索を一時的に解除し、Container 全体の順序で確認してください

### Manual で並び替えたのに Calendar / Kanban / TextLog に反映されない

**症状**: サイドバーで並び替えたのに、Calendar / Kanban / TextLog 側の並びは変わらない。

**結論**: これは **意図的な仕様** です。

**理由**: Manual Order はサイドバー（エントリ一覧）専用の並び順です。Calendar は期日、Kanban はステータス、TextLog はタイムスタンプといった、それぞれのビュー固有の軸で並ぶことを優先しています。

**対処**: サイドバーの表示順を活用したい場合は Detail ビューで操作してください。

### Filter / Search 中に並び替えると挙動が直感と違う

**症状**: フィルタや検索で絞り込んだ状態で ↑ / ↓ を押すと、画面の隣のエントリと入れ替わらないように見える。

**原因**: Manual 並びは常に **Container 全体** の順序として保存されます。フィルタ中でも ↑ / ↓ は「Container 上の直前 / 直後」と入れ替わるため、画面上で非表示のエントリが間に挟まっていると、見た目の並びが 1 手では動いたように見えないことがあります。

**対処**:

- フィルタを一時的に解除してから並び替えると、期待どおりの挙動になります
- フィルタを維持したい場合は、反映先が Container 全体であることを意識して複数回 ↑ / ↓ を押してください

### 種別フィルタに目的の種別（Todo / File など）が見当たらない

**症状**: サイドバーの種別フィルタバーに Todo や File のボタンが表示されない。

**原因**: 種別フィルタは 2 段構成（Primary / Secondary）になっています。Todo / File（attachment）/ Form / Generic / Opaque は **Secondary 段** に配置されており、既定では折りたたまれて非表示です。

**対処**: フィルタバー右端の **▼** ボタンを押して Secondary 段を展開してください。展開状態はフィルタをクリアしても維持されます。

### 種別フィルタで複数選んだら結果が増えた

**症状**: 種別フィルタで Text と Folder を両方選んだら、Text だけのときよりエントリが増えた。

**原因**: これは **意図的な仕様** です。複数の種別を選ぶと **OR 結合**（どれかに該当すれば表示）で動作します。「Text のみ」より「Text または Folder」の方が対象エントリが多くなるのは正常な動作です。

**対処**: 特定の種別だけに絞りたい場合は、不要な種別のボタンを再度押して OFF にしてください。

### 種別フィルタで何も選んでいないのに全エントリが出る

**症状**: 種別フィルタのどのボタンも active になっていないのに、サイドバーにすべてのエントリが表示されている。

**原因**: 種別を 1 つも選んでいない状態は **全表示** として扱われます。「何も選ばない → 何も表示しない」ではありません。これは初期状態や "All" ボタン押下後と同じ状態で、archetype によるフィルタリングが行われていないことを意味します。

**対処**: 特定の種別だけに絞りたい場合は、目的の種別ボタンを押して ON にしてください。

### × クリアを押したらフィルタが全部消えた

**症状**: 検索欄横の **×** をクリックしたら、種別フィルタだけでなくテキスト検索やタグフィルタもすべて解除された。

**原因**: × ボタンは **CLEAR_FILTERS**（全フィルタ一括リセット）です。テキスト検索・種別フィルタ・タグフィルタの 3 つを同時に初期状態に戻します。ただし Secondary 段の展開状態は維持されます。

**対処**: 種別フィルタだけ解除したい場合は × ではなく **All** ボタンを押してください。テキスト検索やタグフィルタはそのまま残ります。

### リビジョン履歴パネルが見当たらない

**症状**: メタパネルに **Revision history** の折りたたみパネルが表示されない。

**考えられる原因**:

1. **選択中のエントリにリビジョンが 0 件** — リビジョンはエントリを編集して確定するたびに追加されます。一度も編集していないエントリには履歴パネル自体が出ません
2. **エントリを選択していない** — サイドバーでエントリをクリックして選択状態にしてください
3. **メタパネルが折りたたまれている** — 右端の META バーをクリックして展開してください

### Restore / Restore as branch のボタンが表示されない

**症状**: Revision history パネルは見えているが、各行の **Restore** / **Restore as branch** ボタンが出ない。

**考えられる原因**: UI 側のガードとバックエンド側の reducer ガードは同じ条件です。以下のいずれかに該当するとボタンは非表示になり、仮に直接 dispatch しても reducer 側で拒否されます。

- **readonly ワークスペース**（「Readonly モードで編集できない」を参照）
- **view-only（埋め込みコンテナから起動）**（「HTML を開いて編集したのに、保存されず消えた」を参照）
- **編集モード中** — いったん Save / Cancel で編集を抜けてから操作してください
- **Import Preview / Batch Import Preview を開いている** — Confirm / Cancel でプレビューを閉じてから操作してください

### Restore as branch しても元のエントリが消えない

**症状**: 履歴パネルで **Restore as branch** を押しても、元のエントリがサイドバーに残り続けている。

**結論**: これは **意図的な仕様** です。

**理由**: Restore as branch は「リビジョンの内容で **新しいエントリ** を追加する」操作で、元エントリは一切変更しません。元エントリと新エントリは `kind: "provenance"` のリレーションで紐付けられ、どちらから派生したかは後から追跡できます（用語集の **Provenance Relation** を参照）。

**対処**:

- 元エントリをもう使わない場合は、通常の削除操作で手動で消してください
- 「復元でその場で上書きして履歴だけ残したい」場合は **Restore as branch** ではなく **Restore（上書き復元）** を使ってください

### 最新 Revert と履歴パネルの Restore の違いが分からない

**症状**: メタパネル上段の **Revert** と、Revision history パネル内の **Restore** の使い分けが分からない。

**対処**: 2 つは動作対象が違うだけで、中身は同じ「選んだリビジョンで本文を上書きする」操作です。

- **Revert**（メタパネル上段）— **最新リビジョン 1 件** を対象にしたショートカット。1 ステップ戻したいだけの典型ケースで素早く使えます
- **Revision history パネルの Restore** — **任意の過去リビジョン** を対象にできます。複数世代さかのぼって戻したいとき、特定日時のスナップショットを取り戻したいときに使います
- **Revision history パネルの Restore as branch** — 戻すのではなく **新しいエントリとして分岐** させます。現在の本文も残しておきたい場合に使います

履歴パネル側の Restore を押したときの挙動は Revert と同じで、上書き直前の状態は新しいリビジョンとして自動保存されます。

### 保存しようとしたら「別のセッションでこのエントリが更新されました」と出た

**症状**: `Ctrl+S` / `Cmd+S` や保存ボタンを押したら、**「別のセッションでこのエントリが更新されました」** というダイアログが出て、保存が完了しない。

**原因**: 編集モードに入ってから保存しようとするまでの間に、**同じエントリが別の経路で更新された** ことが検出されました。具体的には次のいずれかです。

- 別ウィンドウ（Entry Window）や別タブで、同じエントリを編集して先に保存した
- 編集中にそのエントリが削除された
- 編集中にそのエントリの archetype が変わった

これは並行編集でどちらかの内容が黙って消えるのを防ぐための安全装置です。詳細は [05 日常操作 — 並行編集の安全装置](05_日常操作.md) を参照してください。

**対処**: ダイアログの 3 つのボタンからどれか 1 つを選びます。

- **Save as branch**（初期フォーカス）— 自分の編集内容を **新しいエントリ** として分岐保存します。相手側の最新状態もそのまま残るので、一番安全な選択肢です
- **Discard my edits** — 自分の編集内容を **破棄** します。相手側の最新状態のみが残ります
- **Copy to clipboard** — 自分の編集内容の本文をクリップボードへコピーします（ダイアログは閉じません）。その後で **Save as branch** か **Discard my edits** を選んでダイアログを閉じてください

### スキャンラインが表示されない

**症状**: メニューで Scanline を On にしたはずなのに、画面に横縞が出ない。

**考えられる原因**:

1. **ライトモードで On にした** — ライトモードではスキャンラインの横縞が非常に薄く、ほぼ見えない場合があります。Dark または System テーマに切り替えると視認しやすくなります
2. **Off ボタンが選択されている** — Scanline 行は **○ Off / ◉ On** の 2 ボタン segmented control です。**◉ On** ボタンが active 表示になっているか確認してください
3. **read-only なコンテナを開いている** — Rehydrate 直後などの **read-only** 閲覧中は Scanline の切替自体はできますが、設定の保存が抑止されるためリロードすると Off に戻ります。通常の編集可能な状態で On にすれば以降は保持されます

### スキャンライン（横縞）を消したい

**症状**: 画面に薄い横縞が表示されていて、消す方法が分からない。

**対処**:

1. ☰ **Menu** を開く
2. **Scanline** 行の **○ Off** ボタンをクリック → 横縞が消えます

設定は `__settings__` エントリに自動保存され、次回リロード時も Off のまま開きます。

### アクセントカラーを変更したい

**対処**:

1. ☰ **Menu** を開く
2. **Accent** 行の **カラーピッカー**（色のついた小さな四角）をクリック
3. ブラウザ標準の色選択ダイアログから好きな色を選ぶ → 即座に UI 全体に反映されます

元のネオングリーンに戻したいときは **Neon Green に戻す** ボタンを押してください。

### アクセントカラーを変えたら一部の見え方が前のままで残っている

**症状**: Accent を青や紫に変えたが、画面の一部（発光エフェクト、ボタンの hover、選択中アイテムの文字色など）が以前のグリーン系のまま残る。

**原因**: 現在の v1 では `--c-accent` のみがユーザー設定で書き換えられます。次の箇所は**意図的に**追従しない設計です:

- **glow / text-shadow / box-shadow** — 装飾的な発光エフェクトはネオングリーン固定（実害は軽微）
- **アクセント上の文字色**（`--c-accent-fg`）— 既定のダーク色のまま。極端に暗いアクセント色を選ぶと選択中アイテムの文字が読みにくくなる場合があります
- **一部の薄い hover 背景** — 既定のグリーン系のティントが残ります

**迷ったら**: **Neon Green に戻す** ボタンですぐに既定配色へ復帰できます。

### Accent color が保存されない

**症状**: Accent を変更したあと、ブラウザのタブを閉じて再度開いたらネオングリーンに戻っている。

**考えられる原因**: **read-only なコンテナを閲覧中**の可能性があります。Rehydrate 直後など書き込めない状態で Accent を変更しても、UI には即時反映されますが `__settings__` エントリへの保存が抑止されるためリロードで既定色に戻ります。

**対処**: 通常の編集可能な状態（Rehydrate 済み、または IDB 由来のコンテナ）で Accent を変更してください。その場合は自動的に `__settings__` に保存され、次回リロード後も維持されます。

### 外観設定（Theme / Scanline / Accent）がタブ/環境をまたいで引き継がれない

**症状**: 別のタブで開いている PKC2、または別環境にエクスポートした HTML を開くと、設定が既定値に戻っている。

**原因の整理**:

- **同じ環境の別タブで異なる**: 各タブは自分の IndexedDB（または開いているエクスポート HTML の `__settings__`）を読みます。もう一方のタブでの変更はリロードするまで反映されません
- **エクスポート先環境で反映されない**: エクスポート HTML には `__settings__` が同梱されます。エクスポート**以降**に行った設定変更はエクスポート済み HTML には届きません（その HTML はスナップショットなので）
- **Light export（添付なし）で開いた場合**: Light export 由来のビューは `__settings__` も含め **保存が抑止** されます（read-only 扱い）。変更はセッション内だけで消えます

### Kanban 列へのドラッグ時の背景色が変わった

**症状**: Todo カードを Kanban の列にドラッグしたとき、列の背景が以前よりも薄い緑になり、カードのテキストが読めるようになった。

**原因**: これは **意図的な改善** です。以前は列の背景がアクセントカラー（ネオングリーン、100% 不透明）で塗りつぶされており、カード本文が読めなくなることがありました。現在は半透明（約 12% 不透明度）に変更し、破線枠でドロップ先を示す表示に改善されています。

**注意**: サイドバーのフォルダへのドラッグ時のハイライトは従来どおりです（Kanban 列専用の変更です）。

### Kanban カードを選択したときの見え方が前と違う

**症状**: 以前は Kanban カードを選択するとカード全体がアクセントカラーで塗りつぶされていた。現在は薄いティント + 枠線 + シャドウになっている。

**原因**: これは **意図的な可読性改善** です。以前は背景がベタ塗りで本文の文字が読めなくなることがあったため、Kanban カードのみ次の表示に変更しました:

- 背景: アクセントの**薄いティント**（約 14% 不透明度）
- 文字色: 通常色のまま保持
- 選択状態の手がかり: アクセント色の**枠線 + インセットシャドウ + 発光**

サイドバー / Calendar / Slash メニュー等、他のビューの選択表示は従来どおりです（アクセントべた塗り）。

### 各設定項目は独立しているか

**回答**: はい、**11 項目すべてが完全に独立**です。いずれもコンテナ内の `__settings__` エントリに保存され、リロード後も維持されます。

- **Theme** — 配色全体（背景・前景）。☰ Menu の **Theme** 行
- **Scanline** — スキャンライン横縞の表示/非表示。☰ Menu の **Scanline** 行
- **Accent color** — アクセントカラー。☰ Menu の **Accent** 行
- **Border color** — ボーダーカラー。☰ Menu の **Border** 行
- **Background color** — 背景色（全 pane に反映）。☰ Menu の **Background** 行
- **UI Text color** — UI テキストカラー。☰ Menu の **UI Text** 行
- **Body Text color** — 本文テキストカラー。☰ Menu の **Body Text** 行
- **Font** — フォントファミリー（プルダウン + 直接入力）。☰ Menu の **Font** 行
- **Language** — `html[lang]` 属性。☰ Menu の **Language** 行
- **Timezone** — タイムゾーン設定。☰ Menu の **Timezone** 行

どれを変更しても他の項目に影響しません。Menu は画面の高さに応じてスクロールし、画面幅 720px 以上では 2 列に表示されます。

### 保存される設定の一覧

**回答**: FI-Settings v2 では **11 項目すべてが UI から変更可能かつ自動保存** されます。

| 設定 | UI | 保存 |
|---|---|---|
| Theme（Light/Dark/System） | segmented control | **あり** |
| Scanline（Off/On） | segmented control | **あり** |
| Accent color | カラーピッカー + reset | **あり** |
| Border color | カラーピッカー + reset | **あり** |
| Background color | カラーピッカー + reset | **あり** |
| UI Text color | カラーピッカー + reset | **あり** |
| Body Text color | カラーピッカー + reset | **あり** |
| Font | プルダウン + 直接入力 | **あり** |
| Language | プルダウン | **あり** |
| Timezone | プルダウン | **あり** |

保存先は `__settings__` という隠しエントリで、コンテナと一緒にエクスポート/インポートされます。

### Timezone を変えても画面上の日時が変わらない

**症状**: Timezone を Asia/Tokyo に設定したが、サイドバーやカレンダーの日時表示がブラウザのローカル時間のまま。

**原因**: v1 の Timezone 設定は**保存・復元の基盤**のみ実装されています。画面上のすべての日時表示を指定タイムゾーンで書き換える機能は今後のリリースで対応予定です。現時点では設定値は `__settings__` に保存されますが、表示への反映は限定的です。

### Font を変えたらレイアウトが崩れた

**症状**: Font を Sans Serif や Monospace に変えたら、一部の UI パーツでテキストがはみ出す。

**対処**: **System Default** に戻してください。PKC2 のレイアウトは既定のフォントスタック（BIZ UDGothic + Share Tech Mono 等）に最適化されています。大幅に異なるフォントファミリーを選ぶと、文字幅の違いにより一部表示が崩れる場合があります。

### 並行編集ダイアログが Escape で閉じない

**症状**: 「別のセッションでこのエントリが更新されました」のダイアログで `Escape` を押しても、ダイアログの外側をクリックしても閉じない。

**結論**: これは **意図的な仕様** です。

**理由**: 誤って閉じてしまうと編集内容がどこにも残らなくなる可能性があるため、ダイアログは **Save as branch** / **Discard my edits** のどちらかを明示的に選ばないと閉じない設計になっています。**Copy to clipboard** を押してもダイアログ自体は残ります（コピー用途のため、クローズ動作と分離されています）。

**対処**: 編集内容を残したい場合は **Save as branch** を、捨ててよい場合は **Discard my edits** を押してください。どちらを選んでも相手側の最新エントリが上書きされることはありません。

### Save as branch を押したら元のエントリが消えるのが心配

**症状**: 並行編集ダイアログで **Save as branch** を選ぼうとしたが、元のエントリが削除されてしまわないか不安。

**結論**: 元のエントリは **削除も上書きもされません**。

**理由**: Save as branch は「自分の編集内容を **新しいエントリ** として追加する」操作で、相手側が先に保存した最新の状態はそのまま残ります。元エントリと新エントリは `kind: "provenance"` のリレーション（`conversion_kind: "concurrent-edit"`）で紐付けられ、**どちらが元でどちらが並行編集から派生したか** を後から追跡できます。詳しくは用語集の **Provenance Relation** を参照してください。

**対処**:

- 元エントリをもう使わない場合は、ダイアログを閉じた後に通常の削除操作で手動で消してください
- 相手側の内容が不要で自分の編集内容だけ残したい場合は、Save as branch でいったん分岐保存してから、元のエントリを手動で削除するのが安全です

## 用語集

### Bare URL

リンクのラベルと URL が同一であるとき、`[URL](URL)` の冗長形を作らず URL 文字列だけを挿入する形式。PKC2 のリンク貼り付け機能でアドレスバーから URL をコピーした場合などに自動で適用されます。表示時は後述の **Autolink fallback** により `<a>` に描画され、保存データは URL 文字列のまま残ります。

### Autolink fallback（表示時の bare URL リンク化）

本文に含まれる bare URL（`https://` / `http://` / `ftp://` で始まる URL）を、表示レンダリング時にクリック可能な `<a>` に変換する仕組み。保存される本文は URL 文字列のまま変わらず、編集モードの textarea でも plain text として表示されます。クリップボード上にリンクの HTML 情報が無いブラウザ（例: Firefox macOS のアドレスバーコピー）でも、結果として read モードではクリックできる状態になります。`javascript:` / `file://` / `data:text/html` などの危険スキームはこの autolink の対象外で、テキストのまま表示されます。

### Backlink（バックリンク）

他のエントリの本文に `entry:<lid>` 形式で参照されているとき、参照先エントリから見た「被参照元」のこと。meta pane の **Backlinks** セクションに一覧表示されます。Relations（structural / categorical / semantic など）は Backlink に含まれません。

### Broken link（ブロークンリンク）

本文内の `entry:<lid>` で参照している先のエントリが Container に存在しない状態。meta pane の **Outgoing links** に broken マーカー付きで表示され、クリックしても遷移しません。参照先エントリの再作成か、本文の `entry:<lid>` 記述を修正することで解消できます。

### Archetype（アーキタイプ / エントリ種別）

エントリの種類。`text` / `textlog` / `todo` / `form` / `attachment` / `folder` / `generic` / `opaque` の 8 種類があります。archetype ごとに専用のレンダラ（presenter）が本文の見え方を決めます。

### Archetype filter（種別フィルタ）

サイドバー上部のフィルタバーで、表示するエントリを archetype で絞り込む機能。複数の種別を同時に選択でき（OR 結合）、何も選ばない状態は全表示。Primary 段（Text / Textlog / Folder）は常時表示、Secondary 段（Todo / File / Form / Generic / Opaque）は展開ボタンで表示切替。テキスト検索やタグフィルタとは AND で併用できます。詳しくは [05 日常操作 — サイドバーの種別フィルタ](05_日常操作.md) を参照。

### Asset（アセット）

ファイル添付の実体データ。`container.assets` という独立したストアに base64 エンコードされて保存されます。エントリの本文には `asset_key` だけが保存され、実体は分離されています。

### Asset link（アセットリンク / 内部リンク）

添付ファイルを本文から参照するための Markdown 形式リンク。`![ファイル名](asset:キー)`（画像）または `[ファイル名](asset:キー)`（非画像）の形で本文に書かれます。ビューア表示時、画像は inline 展開、非画像はクリック可能なリンクとして表示されます。スクリーンショット貼り付け・DnD 添付・ボタン添付のいずれの経路でも、編集中に自動挿入されます（詳しくは [05 日常操作 — 編集中のドラッグ&ドロップ / ボタン添付](05_日常操作.md) を参照）。

### Staged Render（段階的表示）

画像の多い TEXTLOG で初期表示を軽くするため、ログ行を段階的に描画する仕組み。閲覧モードに入った直後は先頭 8 件のログだけが即時描画され、9 件目以降は placeholder として保留されます。TEXTLOG 閲覧モードのみ対象で、編集モードや他 archetype には影響しません。詳しくは [05 日常操作 — TEXTLOG の画像の段階的表示](05_日常操作.md) を参照。

### Placeholder（プレースホルダ）

Staged Render で描画が保留されているログ行の空カード。タイムスタンプ・重要フラグ・行リンクボタンだけが描画され、画像と本文テキストは未描画です。最低高さ 160 px を確保しており、スクロール位置のズレを抑えます。viewport に近づくと自動的に実体化（hydrate）します。

### Hydrate（ハイドレート / 実体化）

placeholder を画像・本文込みの完全なログ行に置き換える処理。scroll 入場（viewport 下端まで約 400 px）・先読み（先頭 8 件に続く 4 件）・`beforeprint`（印刷）の 3 契機で発生します。一度 hydrate したログは保持され、再スクロールで placeholder に戻ることはありません。

### Image Intake Optimization（画像自動最適化）

512 KB 以上の対応画像（JPEG / PNG / WebP / BMP）を貼り付け・ドロップ・ボタン添付したときに、保存前に WebP 変換 + 長辺 2560 px 縮小を行う機能。確認ダイアログで「最適化して保存」「元のまま保存」を選べる。「次回から確認しない」で **Remembered choice** として記憶でき、操作経路（paste / drop / attach）ごとに独立して管理される。透過 PNG・GIF・SVG・512 KB 未満の画像・Canvas 変換失敗時は自動スキップされ、原画がそのまま保存される。詳しくは [05 日常操作 — 画像の自動最適化](05_日常操作.md) を参照。

### Optimized（最適化済み）

画像自動最適化が適用された添付画像の状態。添付の body JSON に `optimized` フィールドとして変換元の MIME・サイズ・変換方法・画質・リサイズ有無・元寸法・変換後寸法が記録される（provenance metadata）。原画保持を選んだ場合は `original_asset_key` も含まれる。

### Original kept（原画保持）

画像自動最適化の確認ダイアログで「原画も保持する」チェックを ON にした場合に、最適化後の画像に加えて原画も `{asset_key}__original` としてアセットストアに保存される状態。エクスポート時に両方が含まれる。

### Remembered choice（記憶設定）

画像自動最適化の確認ダイアログで「次回から確認しない」を選んだ場合に localStorage に保存される設定。操作経路（paste / drop / attach）ごとに独立して記憶され、一方の経路の記憶が他方に影響することはない。解除するにはブラウザの DevTools から該当の localStorage キー（`pkc2.imageOptimize.preference.{surface}`）を手動削除する。

### Batch Import（一括取り込み）

複数のテキスト / Markdown ファイルをまとめて PKC2 に **追加** で取り込む操作。既存データは置換されません。プレビューで取り込み対象と配置先フォルダを選べます。

### Container（コンテナ）

PKC2 の永続データの最上位の集約。`meta` / `entries` / `relations` / `revisions` / `assets` の 5 要素から構成されます。Export の単位でもあります。

### Context Menu（コンテキストメニュー）

サイドバーでエントリを右クリックしたときに出る小メニュー。編集・複製・削除・別ウィンドウ表示・selected-only export などに最短距離でアクセスできます。

### Current Entry（現在のエントリ）

いまサイドバーで選択していて、詳細ペインに表示されているエントリのこと。Find & Replace や編集中の入力補助はすべてこの「現在のエントリ」に限定されます。

### Current Log（現在のログ行）

TEXTLOG の編集モードで、**`🔎` ボタンを押して対象にしたログ 1 件** のこと。textlog のログ行ごとの Find & Replace は、ダイアログを開いた瞬間にこの「現在のログ行」が決まり、閉じるまで対象は変わりません。別のログ行を対象にしたいときは、いったん Close して別の `🔎` を押します。

### Duplicate Detection（重複検知）

ファイルをドロップした際に、既存の添付と同一内容かどうかを判定する機能（FI-04 G-2）。ファイルの base64 ハッシュ値**とファイルサイズの両方**が既存 attachment と一致した場合のみ「重複」と判定し、情報トーストを表示します。重複と判定されても attachment エントリの作成は必ず実行されます（添付をキャンセルする機能ではありません）。どちらか一方しか一致しない場合は重複とみなしません（誤判定を避けるための安全設計）。

### Persistent Drop Zone（常設ドロップゾーン）

サイドバー最下部に常時表示される「📎 Drop files here」エリア（FI-04 G-3）。ready フェーズでのみアクティブで、センターペインのドロップゾーンと同じ動作でファイルを添付できます。editing フェーズでは非アクティブ（薄く表示、pointer-events: none）になり、editing 専用の経路と競合しません。

### Drag & Drop（DnD）

マウスでエントリを掴んで別の位置にドロップする操作。サイドバーではフォルダ移動、Kanban ではステータス変更、Calendar では期日変更に使えます。複数選択中はまとめて移動できます。

### Entry（エントリ）

データの基本単位。`lid`（ローカル ID）、`title`、`body`、`archetype`、`created_at`、`updated_at` の 6 フィールドを持ちます。旧称は Record でしたが、TypeScript の組み込み型 `Record<K, V>` との衝突を避けて Entry に改名されました。

### Entry Conflict（エントリ衝突）

Import preview の Merge mode 中、host 側と imported 側で **archetype 一致 + 正規化タイトル一致** になる entry の組のこと。内容ハッシュまで一致すれば **C1（content identical）**、内容が違って host 候補が 1 件なら **C2（title matches, content differs）**、host 候補が 2 件以上なら **C2-multi** に分類されます。各 conflict 行で **Keep current** / **Duplicate as branch** / **Skip** のいずれかを選択して解決します。詳しくは [07 保存と持ち出し](07_保存と持ち出し.md) の Merge mode 節を参照。

### Entry Window（別ウィンドウ）

エントリを別のブラウザウィンドウで開いて閲覧・編集する機能。メインワークスペースと並べて参照できます。メイン側の更新は別ウィンドウに自動追従します。

### 並行編集（Concurrent Edit）

同じエントリを **別ウィンドウ** や **複数タブ** などから同時に編集し、両方が保存しようとする状況のこと。PKC2 は編集モードに入った時点のスナップショットと保存時点の状態を比べ、食い違いがあれば後から来た保存を中止して専用ダイアログを出します。ダイアログでは **Save as branch**（新しいエントリとして分岐保存）/ **Discard my edits**（破棄）/ **Copy to clipboard**（クリップボードへ退避、ダイアログは開いたまま）の 3 つから選びます。相手側の最新エントリを黙って上書きしない、自分の編集内容を黙って捨てない、の両方が保証されています。詳しくは [05 日常操作 — 並行編集の安全装置](05_日常操作.md) を参照。

### Export

Container を HTML または ZIP として書き出す機能。配布・バックアップ・マシン移行に使います。範囲（全体 / フォルダ / Selected-only）と形式（HTML Light / HTML Full / ZIP / 単体バンドル）の組み合わせで選びます。

### Find & Replace（検索・置換）

編集中の TEXT エントリの本文内で文字列を検索して一括置換するダイアログ機能。Regex / Case sensitive のオプションがあり、Apply 前に状態行にヒット件数が表示されます。対象は **現在編集中の TEXT 本文のみ** — textlog / form / attachment / folder や readonly / historical は対象外です。

### Primary / Secondary filter group（Primary 段 / Secondary 段）

種別フィルタの 2 段構成のこと。Primary 段には Text / Textlog / Folder が配置されて常時表示、Secondary 段には Todo / File（attachment）/ Form / Generic / Opaque が配置されて既定では折りたたまれています。Secondary 段は展開ボタン（▼ / ▲）で表示切替でき、展開状態はフィルタクリア後も維持されます。

### Plain Search（プレーン検索）

Find & Replace の既定モード（Regex チェック OFF）。Find 欄の文字列はすべてリテラルとして扱われ、`\d+` などの正規表現メタ文字はそのままの文字として検索されます。置換文字列中の `$1` なども back-reference として解釈されず、文字列としてそのまま挿入されます。

### Regex（正規表現）

Find & Replace の opt-in オプション。Find 欄を JavaScript の `RegExp` として解釈します。置換文字列側で `$&` / `$1` / `$<name>` の back-reference が使えるのは Regex が ON のときだけです。無効な正規表現は Apply ボタンを無効化し、状態行に赤文字でエラーメッセージを表示します。

### Scanline（スキャンライン）

CRT モニターの走査線を模した横縞エフェクト。☰ Menu の **Scanline** 行（**○ Off / ◉ On** の segmented control）で切り替えられます。デフォルトは Off。設定は `__settings__` エントリに保存され、リロード後も維持されます（persistent setting）。テーマ（Light / Dark / System）および Accent color とは完全に独立しており、互いに影響しません。詳しくは [05 日常操作 — 外観のカスタマイズ](05_日常操作.md) を参照。

### Accent color（アクセントカラー）

選択状態・ボタン・フォーカスリング等で使われる強調色。☰ Menu の **Accent** 行のカラーピッカーで変更できます。**Neon Green に戻す** ボタンで既定値（`#33ff66`）に復帰できます。設定は `__settings__` エントリに保存され、リロード後も維持されます（persistent setting）。CSS 変数 `--c-accent` のオーバーライドとして適用され、主要な参照箇所が追従しますが、glow / 一部 hover / アクセント上の文字色は既定配色のまま保持されます（意図的な制限）。

### Theme mode（テーマモード）

画面全体の配色を Light / Dark / System の 3 択で切り替える設定。☰ Menu の **Theme** 行の 3 ボタン segmented control で変更できます。初期値は System（ブラウザ/OS の `prefers-color-scheme` に追従）。設定は `__settings__` エントリに保存され、リロード後も維持されます（persistent setting）。

### Neon Green

PKC2 の既定 accent color（`#33ff66`）。Accent をカスタマイズしたあとに **Neon Green に戻す** ボタンで即座に復帰できます。

### System Settings Entry (`__settings__`)

外観・フォント・言語・タイムゾーン等の設定を保存するための**隠しエントリ**。`lid = __settings__`、`archetype = system-settings` で、サイドバーには表示されません。コンテナの一部として IndexedDB に自動保存され、HTML エクスポート時も同梱されます。そのため設定はワークスペースごと別環境に引き継げます。保存される 8 項目: Theme / Scanline / Accent color / Border color / Text color / Font / Language / Timezone。ファイル仕様は `docs/spec/system-settings-hidden-entry-v1-behavior-contract.md` を参照。

### persistent setting / runtime-only setting

PKC2 の設定には次の 2 種類があります。

- **persistent setting** — `__settings__` エントリに自動保存され、リロード後も維持される設定。8 項目: Theme / Scanline / Accent / Border / Text / Font / Language / Timezone
- **runtime-only setting** — UI の一時状態（サイドバー幅、表示中のビュー、展開中のフォルダなど）で、リロードで初期化されるもの

ただし **read-only なコンテナ**（Rehydrate 直後や Light export 由来）を閲覧中は、persistent setting も保存が抑止されるためリロードで戻ります。これはコンテナを意図せず書き換えないための安全装置です。

### Historical Revision（履歴リビジョン）

Revision（リビジョン履歴）から閲覧専用で開いているスナップショット。編集モードには入れず、`✏️ Edit` / `🔎 Replace` などの編集系 UI は現れません。過去の内容を現在の状態に反映したい場合は **復元** を使います。

### HTML Light / HTML Full

HTML Export の 2 つのモード。Light は添付実体を含めず軽量、Full は全データを gzip 圧縮付きで同梱します。

### Import

HTML または ZIP ファイルから Container を読み込む機能。Import preview 上部のラジオで **Replace**（現在のワークスペースを完全に置き換え）か **Merge**（既存データを保持したまま追加取り込み）を選べます。テキスト / Markdown の単純追加が目的なら Batch Import や単体バンドル（.text.zip / .textlog.zip）も候補です。

### Merge mode（マージ取り込み）

Import preview で選べるモードの片方（既定は Replace）。既存 Container を保持したまま imported 側 entry を追加取り込みします。host / imported で同じ archetype + 正規化タイトルの entry が見つかると **Entry conflict** として抽出され、**Keep current** / **Duplicate as branch** / **Skip** を 1 件ずつ選んで解決します。C2 / C2-multi 行がすべて解決するまで Confirm merge は押せません。詳細は [07 保存と持ち出し](07_保存と持ち出し.md) を参照。

### IndexedDB（IDB）

ブラウザ内の永続ストレージ API。PKC2 はここに Container と Assets を保存します。ブラウザごと・ドメインごとにストアが分かれます。

### LID（Local ID）

Container 内でユニークな Entry の ID。複数の Container 間で衝突する可能性があるため「ローカル」と呼ばれます。

### Manual Order（手動並び替え）

サイドバーの並び順セレクタで **Manual** を選んだときに有効になる、ユーザー定義の並び順。Container の `entry_order` に LID 配列として保存され、他のソートに切り替えても Container には残ります。Detail ビューで選択中のエントリに現れる ↑ / ↓ ボタンで 1 件ずつ前後に移動できます。Calendar / Kanban / TextLog には反映されません。

### Multi-select（複数選択）

`Ctrl+Click` / `Shift+Click` で複数のエントリを同時に選ぶ操作。一括削除、一括フォルダ移動、Kanban / Calendar での一括ステータス変更・期日変更に使えます。

### pkc-data / pkc-meta / pkc-core

PKC2 HTML の固定スロット。それぞれ Container の JSON・リリースメタデータ・アプリケーションコード（JavaScript バンドル）を格納します。

### Preview（プレビュー）

Import / 変換を確定する前の確認画面。Import Preview では読み込み内容を確認してから Confirm を押します。TEXT ↔ TEXTLOG 変換プレビューでは変換結果の構造を確認してから作成を押します。

### Provenance Relation（由来リレーション）

「あるエントリが別のエントリから派生した」という由来を記録する `kind: "provenance"` のリレーション。現在は次の 3 経路で自動付与されます。

- **Merge mode の Duplicate as branch** — 向きは「derived（merge で新採番された新 lid）→ source（対応 host lid）」に固定され、metadata に一致種別（content-equal / title-only / title-only-multi）、検出時刻、imported の title / archetype スナップショットが記録されます。C2-multi の場合は `host_candidates` に全候補 lid も残るため、「merge で分岐した entry がどの host から派生したか」を後から追跡できます
- **Revision history の Restore as branch** — 向きは「source（分岐元の既存 entry lid）→ derived（分岐で新採番された新 lid）」で、metadata に `conversion_kind: "revision-branch"`、対象 `source_revision_id`、（リビジョンに付いている場合のみ）`source_content_hash` が記録されます。分岐したエントリがどのリビジョンから派生したかを後から辿れます
- **並行編集ダイアログの Save as branch** — 向きは「source（相手側が先に保存した既存 entry lid）→ derived（自分の編集内容を新採番して保存した新 lid）」で、metadata に `conversion_kind: "concurrent-edit"`、検出時刻、編集開始時の `source_updated_at`、（付いている場合のみ）`source_content_hash` が記録されます。並行編集で分岐した entry が、どの時点の元エントリから派生したかを後から辿れます

### Readonly

HTML Export の mutability の 1 つ。開いた時に編集ボタンが表示されず、閲覧専用の UI ポリシーになります。セキュリティ機能ではなく、あくまで「うっかり編集を防ぐ」ガードです。

### View-only（埋め込みコンテナからの起動）

Export HTML の埋め込みコンテナ（pkc-data）から起動した状態のこと。編集はメモリ上では可能ですが、IndexedDB への自動保存は抑止されます（タブを閉じれば消えます）。受信者の既存ワークスペースを上書きしないための仕様で、編集を残したい場合は Import を経由して明示的に取り込みます。Readonly と直交する概念で、Readonly Export でかつ埋め込み起動の場合は両方の制約がかかります。

### Rehydrate（再水和）

Readonly HTML を Editable なワークスペースに昇格させる操作。新しい Container ID が発行され、IndexedDB に保存されます。元の HTML ファイルは変更されません。

### Relation（リレーション）

エントリ同士を結ぶタイプ付きのエッジ。`structural` / `categorical` / `semantic` / `temporal` / `provenance` の 5 種類があります。`provenance` は Merge mode の Duplicate as branch と Revision history の Restore as branch で自動付与される由来リレーション（Provenance Relation を参照）。

### Revision（リビジョン）

エントリの編集前の状態を記録したスナップショット。編集のたびに自動保存され、後から復元できます。選択中のエントリに 1 件以上あるとメタパネルに **Revision history** パネル（折りたたみ式）が出て、各行から 2 つの復元操作が選べます。

- **Restore（上書き復元）** — 現在のエントリの本文・タイトルをその行のスナップショットで上書きします。上書き直前の状態も新しいリビジョンとして自動保存されます
- **Restore as branch（分岐復元）** — スナップショットの内容で **新しいエントリ** を追加し、`provenance` リレーションで元エントリと紐付けます。元エントリは変更されません

メタパネル上段の **Revert** ボタンは「最新リビジョンへの Restore」のショートカットとして残っています。

### Selected-only Export（単体エクスポート）

選択中の 1 エントリと、それが参照する添付だけを小さな `.text.zip` / `.textlog.zip` に書き出す操作。別ワークスペースへのピンポイントな移植に使います。

### Toast（トースト通知）

画面の隅に表示される短い非ブロッキング通知。warn（警告）/ info（情報）/ error（エラー）の 3 種類があり、数秒で自動消去されます。ZIP Import 警告はこの warn で表示されます。

### Transclusion（埋め込み）

`![](entry:<lid>)` と書いて、他のエントリの内容を本文内にその場で展開表示する機能。TODO カード埋め込みやドキュメントの相互参照に便利です。循環参照や深い入れ子は自動で遮断されます。

### ZIP Package

`manifest.json` + `container.json` + `assets/` で構成される PKC2 独自の ZIP 形式（拡張子 `.pkc2.zip`）。添付ファイルを生バイナリで保存するため、最もサイズ効率が良く、Git 管理にも向きます。

### Workspace（ワークスペース）

編集可能な状態の Container を指します。IndexedDB に保存されていて、UI 上で編集・作成・削除ができる状態です。Readonly HTML と対比されます。
