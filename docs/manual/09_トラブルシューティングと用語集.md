# トラブルシューティングと用語集

## トラブルシューティング

### データが消えた・見当たらない

**症状**: 前回作成したエントリが表示されない。

**考えられる原因**:

1. **別のブラウザ / 別の HTML ファイルで開いている**: IndexedDB はブラウザごと・URL ごとに分かれています。同じ HTML ファイルを同じブラウザで開き直してください
2. **ブラウザのサイトデータをクリアした**: ブラウザの履歴・サイトデータ削除で IndexedDB も消えます
3. **プライベートブラウジングで使った**: プライベートモードのデータはタブを閉じると消えます
4. **ファイルのダウンロード先が変わった**: 同じ HTML ファイルでも保存場所が変わると `file:///` URL が変わり、別のストアになります

**対処**:

- データの保険として定期的に Export（特に ZIP Package）でバックアップしてください
- 複数デバイスで使う場合は、ZIP Package を共有ストレージに置いて運用するのが確実です

### Readonly モードで編集できない

**症状**: エントリの編集ボタンが表示されない。

**原因**: Readonly としてエクスポートされた HTML を開いています。

**対処**:

- 編集したい場合は **Rehydrate to Workspace** ボタンを押してください
- Rehydrate すると新しい Container ID が発行され、編集可能なワークスペースとして IndexedDB に保存されます
- 元の HTML ファイル自体は変更されません

### ファイルサイズが大きいと警告が出る

**症状**: ファイル添付時に「このファイルは大きすぎます」的なメッセージ。

**原因と対処**:

- **1 MB 以上**: 軽い警告。操作はブロックされません
- **5 MB 以上**: 重めの警告。Export の HTML サイズに影響します。大きい添付が多い場合は **ZIP Package** でのバックアップを検討してください
- これらは制限ではなく情報提供です。無視して進めることもできます

### Export したら HTML が数十 MB になった

**症状**: 書き出した HTML が重い。

**原因**: 添付ファイルを HTML Full モードで同梱すると、gzip 圧縮しても一定のサイズになります。

**対処**:

- 共有用途なら **HTML Light**（添付実体なし）を使う
- バックアップ用途なら **ZIP Package**（base64 膨張なし）を使う
- 巨大な添付は元ファイルを別管理にする選択も検討

### Import したらデータが全部置き換わった

**症状**: Import 後に元のエントリが消えた。

**原因**: HTML / ZIP の **Import は全置換** です。マージ機能はありません。

**対処**:

- Import 前に必ず Export でバックアップを取る習慣をつけてください
- Confirm Import のダイアログで内容を確認してから確定しましょう
- ファイルを単純に **追加** したいだけなら、**Batch Import** や **単体バンドル（.text.zip / .textlog.zip）** を使う方が目的に合います

### ZIP Import 後に「警告あり」の toast が出た

**症状**: `.pkc2.zip` の Import 直後に「ZIP import succeeded with a warning: ...」や「N warnings」といった黄色い toast が出る。

**原因**: ZIP の中身に不整合があり、PKC2 側で安全に解決したことを知らせる通知です。**Import 自体は成功しています**。次のような状況で出ます。

- 同じ asset key が複数回入っていた
- `manifest.json` や `container.json` が複数入っていた
- `/` や `..` を含む不正な asset key があった

**対処**:

- 心当たりのある ZIP（例: 2 つの ZIP を手動でマージしたもの）であれば、多くの場合そのまま進めて大丈夫です
- 詳細を知りたい場合はブラウザのデベロッパーツール（`F12`）のコンソールを開いて `[PKC2] ZIP import warning: ...` の行を確認してください
- 出どころ不明の ZIP の場合は、toast を閉じずに一度内容を確認してから進めるとよいです

### IndexedDB が壊れた・読み込みエラー

**症状**: PKC2 を開いても初期化が終わらない、エラーが出る。

**対処**:

1. ブラウザのデベロッパーツール（`F12`）を開き、コンソールのエラーメッセージを確認
2. **Clear IDB** ボタンが表示されている場合はそれを使って IndexedDB をクリア（データは消えます）
3. IDB クリア後、HTML 内の pkc-data にフォールバックします。Export 済みのバックアップから Import で復元してください
4. 再発する場合は別のブラウザで試してみてください

### ブラウザのタブを閉じたら保存されなかった気がする

**症状**: 編集中にタブを閉じたら最新の変更が消えた。

**対処**:

- PKC2 は 300 ミリ秒のデバウンスで保存するため、編集直後にタブを閉じると間に合わないことがあります
- 確実に保存したい場合は `Ctrl+S` / `Cmd+S` で明示的に確定してください
- `pagehide` イベントでの強制保存もありますが、OS の強制終了では働きません

### 別ウィンドウ（Entry Window）が開かない

**症状**: エントリをダブルクリックしても別ウィンドウが開かない。

**対処**:

- ブラウザのポップアップブロックが有効だと開けません。アドレスバー付近のポップアップ許可アイコンから PKC2 を許可してください
- 別ウィンドウは `window.open` を使います。ブラウザ設定で無効化されているケースでは動きません

### 別ウィンドウが「古い内容」のままになる

**症状**: メインワークスペースで Todo を更新したのに、別ウィンドウ内で表示している埋め込みカードや本文が古いまま。

**対処**:

- 通常は自動で追従します。追従しない稀なケースでは、別ウィンドウをいったん閉じて再度開き直してください
- これは PKC2 の既知の追従経路で、メイン側の更新が別ウィンドウに伝わるようになっています

### 埋め込み（iframe）で PKC2 が動かない

**症状**: 親ページに iframe で PKC2 を埋め込んだが、表示されない・操作できない。

**対処**:

- iframe の `sandbox` 属性に `allow-scripts allow-same-origin` が含まれているか確認
- `allow-top-navigation` は付けないでください（セキュリティ境界を壊します）
- 親ページから `postMessage` で通信する場合は `origin` を検証してください
- 詳細は開発者向け文書 `docs/planning/14_基盤方針追補_clone_embed_message.md` を参照

### Replace（置換）ボタンが見当たらない

**症状**: 編集中の action bar に `🔎 Replace` ボタンが出ない。

**考えられる原因**:

1. **編集モードに入っていない** — まず `✏️ Edit` を押して編集モードに遷移してください
2. **対象エントリが TEXT ではない** — Replace は TEXT エントリの本文に限定です。textlog / form / attachment / folder は対象外
3. **Readonly モードで開いている** — Rehydrate してから編集してください（「Readonly モードで編集できない」を参照）
4. **Historical revision を閲覧中** — 履歴スナップショットは読み取り専用です。現在の状態に反映したい場合は **復元** を使ってください

### 置換ダイアログの Apply が押せない

**症状**: ダイアログを開いて検索文字列を入れたのに、`Apply` ボタンがグレーアウトしたままで押せない。

**考えられる原因**:

- **Find 欄が空** — 検索文字列を入力してください
- **Regex が有効で、正規表現として無効** — 状態行に「Invalid regex: ...」が赤文字で表示されます。パターンを直すか、Regex チェックを外してください
- **ヒット 0 件** — 状態行に「No matches in current entry.」と出ている場合は、本文中に一致する文字列がありません

### 置換ダイアログの Regex で思った通りに置換できない

**症状**: `$1` や `\d+` を期待通り動かしたい。

**対処**:

- **Regex チェックを ON** にしていないと、Find 欄は **リテラル文字列** として扱われます。`\d+` は `\d+` という 3 文字そのものの検索になります
- 置換文字列（Replace with）で `$1` / `$&` / `$<name>` などの back-reference を使うには、**Regex チェックが ON** である必要があります。OFF の時は `$1` はリテラル文字列としてそのまま挿入されます

### リンクをコピーして貼り付けても URL が残らない

**症状**: ブラウザから「`Example`」というリンクをコピーしたはずなのに、本文にはラベルだけが入り URL が消える。

**考えられる原因**:

1. **貼り付け先が TEXT 本文の textarea ではない** — タイトル欄・textlog 追記欄・form は対象外です
2. **編集モードに入っていない** — TEXT エントリを編集モードで開いてから貼り付けてください
3. **クリップボードに text/html が含まれていない** — コピー元がプレーンテキストしか提供していないケースです。この場合はラベルだけが貼り付けられます
4. **URL が危険なスキーム** — `javascript:` / `data:` / `vbscript:` の URL は安全のためリンク化されず、ラベルのみプレーンテキストとして残ります

**対処**: TEXT 本文の編集モードで貼り付け、かつコピー元が text/html 形式のリンクを提供していれば、自動で `[ラベル](URL)` の形に整形されます。

### Export した HTML を開いたら「起動元を選ぶダイアログ」が出た

**症状**: Export HTML を開いた瞬間に「どちらのコンテナを開きますか？」のダイアログが表示される。

**原因**: HTML 内部の埋め込みコンテナ（pkc-data）と、そのブラウザに保存済みの IndexedDB のワークスペースの **両方が見つかった** ためです。どちらを表示するかはユーザーに選ばせる仕様です。

**対処**:

- **Export の中身を見たいだけ**: 「埋め込みを開く (view-only)」を選択。編集しても IDB には保存されません（タブを閉じれば元のワークスペースに戻ります）
- **普段使いのワークスペースを続けたい**: 「IndexedDB を開く」を選択。埋め込みコンテナは無視され、手元の IDB データがそのまま使われます
- このダイアログは Escape や背景クリックでは閉じません。必ずどちらかを選んでください

詳しくは [07 保存と持ち出し — HTML を開いたときの起動元](07_保存と持ち出し.md) を参照してください。

### HTML を開いて編集したのに、保存されず消えた

**症状**: Export HTML を開いて編集したが、タブを閉じてから開き直すと編集内容が消えている。

**原因**: 埋め込みコンテナから起動した（view-only）状態では、**IndexedDB への自動保存が行われません**。これは「受信者が Export を見るだけで自分のワークスペースが上書きされる」事故を防ぐための仕様です。編集は画面上のみで、リロードで消えます。

**対処**:

- そのセッションだけでよい確認用途なら、これは意図した挙動です
- 編集を残したい場合は、**明示的に Import する** 必要があります:
  1. 現在の HTML をブラウザの標準機能（`Ctrl+S` / 右クリック→保存）でファイル保存
  2. **Data…** パネルの **Import** ボタンからそのファイルを選ぶ
  3. プレビュー→ **Confirm Import** で IndexedDB に取り込まれます
- Import 確定後は通常のワークスペースとして自動保存されます

### Export した HTML を開くと、手元の IndexedDB ではなく Export 側の内容が表示される

**症状**: Export HTML を開いた時、いつも使っているワークスペースではなく、Export 時点のスナップショットが表示される。ダイアログは出ない。

**原因**: そのブラウザに PKC2 の IndexedDB がまだ無い（初回 / 別ブラウザ / サイトデータクリア後 など）ため、埋め込みコンテナから起動しています。両方が揃っていればダイアログが出ますが、片側しかない場合は自動的にある方から起動します。

**対処**:

- 通常はこれが期待動作です。Export の中身をそのまま見たい用途に向いています
- この状態は **view-only** なので、編集しても IndexedDB には保存されません。編集を残したい場合は上記「HTML を開いて編集したのに、保存されず消えた」の Import 手順を使ってください
- 普段のワークスペースを別ブラウザで作り直したい場合は、空の HTML（生の `pkc2.html`）を開くか、Export ファイルを Import してください

### TEXTLOG のログ行に `🔎` ボタンが見当たらない

**症状**: textlog を編集モードにしたのに、各ログ行に `🔎` ボタンが現れない。

**考えられる原因**:

1. **編集モードに入っていない** — まず textlog の `✏️ Edit` を押してください。viewer（閲覧モード）では表示されません
2. **Readonly ワークスペースを開いている** — 「Readonly モードで編集できない」を参照
3. **Historical revision を閲覧中** — 履歴スナップショットは読み取り専用で、編集モードに入れません
4. **対象が textlog ではない** — `🔎` ボタンは textlog entry の各ログ行にのみ表示されます。TEXT の本文には action bar 側の `🔎 Replace` ボタンが別途あります

### TEXTLOG のログ行置換ダイアログの Apply が押せない

**症状**: ログ行の置換ダイアログを開いたのに、`Apply` がグレーアウトしている。

**考えられる原因と対処**:

- **Find 欄が空** — 検索文字列を入力してください
- **Regex が有効で、無効な正規表現** — 状態行に `Invalid regex: ...` が赤文字で表示されます。パターンを直すか Regex チェックを外してください
- **ヒット 0 件** — 状態行に `No matches in current log.` と出ている場合、そのログ行の本文中に一致する文字列がありません。他のログ行を対象にしたい場合はいったん Close して別の行の `🔎` を押してください

### TEXTLOG の複数ログをまとめて置換したい

**結論**: v1 では提供していません（意図的な仕様）。

**理由**: ログ単位の独立性を守るためです。時刻・フラグ・並び順・ID を壊すリスクが高く、慎重に設計する必要があるため段階的に解放します。

**対処**:

- 今は各ログ行の `🔎` を 1 件ずつ押して順番に置換してください
- 将来の拡張（複数ログ / 全 textlog 一括置換）は別テーマとして検討されています

### TEXTLOG でログ行を置換したが、時刻やフラグは変わらない

**症状**: ログ行の本文を Apply で置換したが、タイムスタンプや ★フラグ、ログ ID は変わらなかった。

**結論**: これは **意図的な仕様** です。

**理由**: ログ行の `text` のみが置換の対象で、時刻 / フラグ / ID / 並び順 / 他のログ行は構造的に保護されています。時刻やフラグを変更したい場合は、編集モードの該当行の timestamp span（読み取り専用）ではなく、チェックボックスや別 UI を使うか、削除 + 再追記で対応してください。

### TEXT / TEXTLOG の変換プレビューが勝手に閉じた

**症状**: プレビューを開いていたのに、エントリを選び直したり編集モードに入ったら消えた。

**原因**: プレビューは「今選んでいるエントリ」に紐づく一時 UI です。別のエントリを選んだり編集モードに入ると、整合性を保つため自動で閉じられます。

**対処**:

- 変換を完了させたい場合は preview 内で **作成** を先に押してください
- いったん閉じた後に再度開くこともできます（元エントリは変更されません）

## 用語集

### Archetype（アーキタイプ / エントリ種別）

エントリの種類。`text` / `textlog` / `todo` / `form` / `attachment` / `folder` / `generic` / `opaque` の 8 種類があります。archetype ごとに専用のレンダラ（presenter）が本文の見え方を決めます。

### Asset（アセット）

ファイル添付の実体データ。`container.assets` という独立したストアに base64 エンコードされて保存されます。エントリの本文には `asset_key` だけが保存され、実体は分離されています。

### Batch Import（一括取り込み）

複数のテキスト / Markdown ファイルをまとめて PKC2 に **追加** で取り込む操作。既存データは置換されません。プレビューで取り込み対象と配置先フォルダを選べます。

### Container（コンテナ）

PKC2 の永続データの最上位の集約。`meta` / `entries` / `relations` / `revisions` / `assets` の 5 要素から構成されます。Export の単位でもあります。

### Context Menu（コンテキストメニュー）

サイドバーでエントリを右クリックしたときに出る小メニュー。編集・複製・削除・別ウィンドウ表示・selected-only export などに最短距離でアクセスできます。

### Current Entry（現在のエントリ）

いまサイドバーで選択していて、詳細ペインに表示されているエントリのこと。Find & Replace や編集中の入力補助はすべてこの「現在のエントリ」に限定されます。

### Current Log（現在のログ行）

TEXTLOG の編集モードで、**`🔎` ボタンを押して対象にしたログ 1 件** のこと。textlog のログ行ごとの Find & Replace は、ダイアログを開いた瞬間にこの「現在のログ行」が決まり、閉じるまで対象は変わりません。別のログ行を対象にしたいときは、いったん Close して別の `🔎` を押します。

### Drag & Drop（DnD）

マウスでエントリを掴んで別の位置にドロップする操作。サイドバーではフォルダ移動、Kanban ではステータス変更、Calendar では期日変更に使えます。複数選択中はまとめて移動できます。

### Entry（エントリ）

データの基本単位。`lid`（ローカル ID）、`title`、`body`、`archetype`、`created_at`、`updated_at` の 6 フィールドを持ちます。旧称は Record でしたが、TypeScript の組み込み型 `Record<K, V>` との衝突を避けて Entry に改名されました。

### Entry Window（別ウィンドウ）

エントリを別のブラウザウィンドウで開いて閲覧・編集する機能。メインワークスペースと並べて参照できます。メイン側の更新は別ウィンドウに自動追従します。

### Export

Container を HTML または ZIP として書き出す機能。配布・バックアップ・マシン移行に使います。範囲（全体 / フォルダ / Selected-only）と形式（HTML Light / HTML Full / ZIP / 単体バンドル）の組み合わせで選びます。

### Find & Replace（検索・置換）

編集中の TEXT エントリの本文内で文字列を検索して一括置換するダイアログ機能。Regex / Case sensitive のオプションがあり、Apply 前に状態行にヒット件数が表示されます。対象は **現在編集中の TEXT 本文のみ** — textlog / form / attachment / folder や readonly / historical は対象外です。

### Plain Search（プレーン検索）

Find & Replace の既定モード（Regex チェック OFF）。Find 欄の文字列はすべてリテラルとして扱われ、`\d+` などの正規表現メタ文字はそのままの文字として検索されます。置換文字列中の `$1` なども back-reference として解釈されず、文字列としてそのまま挿入されます。

### Regex（正規表現）

Find & Replace の opt-in オプション。Find 欄を JavaScript の `RegExp` として解釈します。置換文字列側で `$&` / `$1` / `$<name>` の back-reference が使えるのは Regex が ON のときだけです。無効な正規表現は Apply ボタンを無効化し、状態行に赤文字でエラーメッセージを表示します。

### Historical Revision（履歴リビジョン）

Revision（リビジョン履歴）から閲覧専用で開いているスナップショット。編集モードには入れず、`✏️ Edit` / `🔎 Replace` などの編集系 UI は現れません。過去の内容を現在の状態に反映したい場合は **復元** を使います。

### HTML Light / HTML Full

HTML Export の 2 つのモード。Light は添付実体を含めず軽量、Full は全データを gzip 圧縮付きで同梱します。

### Import

HTML または ZIP ファイルから Container を読み込む機能。現在のワークスペースを**完全に置き換え**ます。追加だけしたい場合は Batch Import や単体バンドル（.text.zip / .textlog.zip）を使ってください。

### IndexedDB（IDB）

ブラウザ内の永続ストレージ API。PKC2 はここに Container と Assets を保存します。ブラウザごと・ドメインごとにストアが分かれます。

### LID（Local ID）

Container 内でユニークな Entry の ID。複数の Container 間で衝突する可能性があるため「ローカル」と呼ばれます。

### Multi-select（複数選択）

`Ctrl+Click` / `Shift+Click` で複数のエントリを同時に選ぶ操作。一括削除、一括フォルダ移動、Kanban / Calendar での一括ステータス変更・期日変更に使えます。

### pkc-data / pkc-meta / pkc-core

PKC2 HTML の固定スロット。それぞれ Container の JSON・リリースメタデータ・アプリケーションコード（JavaScript バンドル）を格納します。

### Preview（プレビュー）

Import / 変換を確定する前の確認画面。Import Preview では読み込み内容を確認してから Confirm を押します。TEXT ↔ TEXTLOG 変換プレビューでは変換結果の構造を確認してから作成を押します。

### Readonly

HTML Export の mutability の 1 つ。開いた時に編集ボタンが表示されず、閲覧専用の UI ポリシーになります。セキュリティ機能ではなく、あくまで「うっかり編集を防ぐ」ガードです。

### View-only（埋め込みコンテナからの起動）

Export HTML の埋め込みコンテナ（pkc-data）から起動した状態のこと。編集はメモリ上では可能ですが、IndexedDB への自動保存は抑止されます（タブを閉じれば消えます）。受信者の既存ワークスペースを上書きしないための仕様で、編集を残したい場合は Import を経由して明示的に取り込みます。Readonly と直交する概念で、Readonly Export でかつ埋め込み起動の場合は両方の制約がかかります。

### Rehydrate（再水和）

Readonly HTML を Editable なワークスペースに昇格させる操作。新しい Container ID が発行され、IndexedDB に保存されます。元の HTML ファイルは変更されません。

### Relation（リレーション）

エントリ同士を結ぶタイプ付きのエッジ。`structural` / `categorical` / `semantic` / `temporal` の 4 種類があります。

### Revision（リビジョン）

エントリの編集前の状態を記録したスナップショット。編集のたびに自動保存され、後から復元できます。復元は「その内容で現在を上書き」する操作で、復元前の状態も別リビジョンとして残ります。

### Selected-only Export（単体エクスポート）

選択中の 1 エントリと、それが参照する添付だけを小さな `.text.zip` / `.textlog.zip` に書き出す操作。別ワークスペースへのピンポイントな移植に使います。

### Toast（トースト通知）

画面の隅に表示される短い非ブロッキング通知。warn（警告）/ info（情報）/ error（エラー）の 3 種類があり、数秒で自動消去されます。ZIP Import 警告はこの warn で表示されます。

### Transclusion（埋め込み）

`![](entry:<lid>)` と書いて、他のエントリの内容を本文内にその場で展開表示する機能。TODO カード埋め込みやドキュメントの相互参照に便利です。循環参照や深い入れ子は自動で遮断されます。

### ZIP Package

`manifest.json` + `container.json` + `assets/` で構成される PKC2 独自の ZIP 形式（拡張子 `.pkc2.zip`）。添付ファイルを生バイナリで保存するため、最もサイズ効率が良く、Git 管理にも向きます。

### Workspace（ワークスペース）

編集可能な状態の Container を指します。IndexedDB に保存されていて、UI 上で編集・作成・削除ができる状態です。Readonly HTML と対比されます。
